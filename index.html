1. الفلسفة البرمجية لمحرك TPAE

Tivoli's Process Automation Engine (TPAE) ليس برنامجاً مستقلاً Not a standalone product، بل هو منصة أساسية مشتركة Common Platform Layer.

💡 التشبيه البرمجي
مثلما يوفر لك إطار العمل Laravel الخدمات الأساسية (الـ ORM، والـ Authentication، والـ Routing، والـ Queueing) لتبني فوقه متجراً إلكترونياً أو نظام تذاكر، فإن TPAE يوفر نفس الخدمات الأساسية لتبني فوقها شركة IBM منتجاتها العملاقة مثل Maximo للمنشآت أو Control Desk لقطاع الـ IT.

2. الميزة القاتلة: المعمارية القائمة على البيانات الوصفية

Metadata-Driven Architecture

في هندسة البرمجيات التقليدية، إذا أردت إضافة حقل Attribute جديد، فإنك تعدل الكود والـ Database Schema يدوياً. في TPAE الأمر مختلف تماماً:

  • يتم تخزين تصاميم الشاشات، وعلاقات الجداول، وقواعد التحقق Validation Rules على هيئة بيانات وصفية (Metadata) داخل جداول النظام System Tables.
✅ الهدف الهندسي: Upgradeable Customizations
عندما تقوم الشركة بترقية النظام (System Upgrade)، تقوم IBM باستبدال ملفات الكود المصدري الأساسي للنظام Core Engine Code، بينما تظل جداول الميتا-داتا الخاصة بتخصيصات شركتك كما هي دون مساس، مما يمنع كسر الكود (Breaking Changes) ويوفر ملايين الدولارات في عمليات الصيانة.

3. موديولات النظام (System Modules)

يجمع النظام التطبيقات ذات الصلة الوظيفية داخل وحدات منطقية Modules، وهو ما يشبه الـ Namespaces أو الـ Packages في لغات البرمجة:

🏭
Asset (الأصول)
تتبع المواقع (Locations) والمعدات طوال دورتها الحياتية.
🔧
Work (العمل)
تتبع خطط العمل وطلبات وأوامر الصيانة (Work Orders).
🎧
Service (الخدمات)
إدارة طلبات الخدمة (Service Requests) واتفاقيات مستوى الخدمة (SLAs).
📝
Contract (العقود)
إدارة عقود الشراء، الإيجار، الضمان، والعمالة.
📦
Inventory & Procurement
تتبع حركات المخازن ودورة الشراء من طلب الاقتباس (RFQ) وحتى أمر الشراء (PO) واستلام الفواتير.

4. أنواع التطبيقات من حيث التصنيف البرمجي

Application Types

⚙️
Core Applications
تطبيقات مركزية مفعّلة في أي منتج يعتمد على TPAE، وهي إلزامية لتقديم خدمات المنصة الأساسية (مثل تطبيق Classifications وتطبيق Security Groups).
🔗
Shared Applications
تطبيقات لا يتطلبها عمل المنصة بشكل إلزامي، ولكن يتم مشاركتها واستخدامها من قِبل منتجات متعددة (مثل Workflow Designer وتطبيق Persons).
🎯
Product-Specific Applications
تطبيقات تُكتب خصيصاً لتنفيذ وظائف فريدة داخل منتج محدد (مثل تطبيق الصيانة الوقائية Preventative Maintenance في ماكسيمو).

5. المكونات الأربعة الرئيسية ومعمارية الطبقات

The Four Key Components — Multi-Layered Architecture

يتكون المحرك داخلياً من أربع طبقات معمارية مترابطة:

🖥️
أ. واجهة المستخدم المشتركة
Common User Interface — واجهة تعتمد بالكامل على المتصفح (Web-based). تتميز بمعمارية Zero-Client Footprint، أي أنه لا يتم تحميل أو تثبيت أي كود عميل (لا يوجد Java Applets أو Active-X) على جهاز المستخدم. وتتغير الواجهة بناءً على دور المستخدم Role-based UI.
🔧
ب. خدمات التكوين المشتركة
Common Configuration Services — تتيح للمطورين والمديرين تعديل واجهات المستخدم UI Designer وتوسيع حقول قاعدة البيانات من داخل السيستم نفسه دون الحاجة لكتابة كود برمجي خارجي.
🗄️
ج. نظام البيانات الفرعي المشترك
Common Data Subsystem — يعتمد على نموذج بيانات موحد ومعزول عن نوع قاعدة البيانات Database-Agnostic، حيث يقوم بترجمة الأوامر برمجياً لتعمل بسلاسة سواء كانت IBM DB2, Oracle, أو MS SQL Server.
⚡
د. محرك سير العمل والأتمتة
Process Workflow Runtime — المحرك المسؤول عن إدارة التعاون بين المستخدمين، إرسال الإشعارات Notifications، وإطلاق محرك التصعيد Escalation Engine المجدول زمنياً لتنفيذ إجراءات تلقائية عند تأخر المعاملات.
🗂️
الهيكل التنظيمي ومستويات عزل البيانات
Data Hierarchy Levels • 4 مستويات هرمية
▼

لكي تفهم كيف المحرك بيعمل عزل للبيانات Data Isolation لتقديم ميزة الـ Multi-Tenancy في الشركات الكبيرة، يقسم الكتاب البيانات إلى 4 مستويات هرمية صارمة:

  1. مستوى النظام (System Level): أعلى مستوى في الهرم، وتكون البيانات هنا عامة ومتاحة لجميع الكيانات (مثل ملفات تعريف المستخدمين والمجموعات الأمنية).
  2. مستوى المجموعات (Set Level): يتيح مشاركة البيانات العامة بين عدة مؤسسات لتوفير وقت الإدخال.
    • مجموعة الأصناف (Item Set): كتالوج موحد لتعريفات قطع الغيار والمواد.
    • مجموعة الشركات (Company Set): دليل موحد للموردين والمصنعين المعتمدين.
  3. مستوى المؤسسة (Organization Level): يمثل كياناً قانونياً ومالياً مستقلاً Legal Entity.
    ⚠️ شرط كتابي صارم
    كل ما يقع تحت المؤسسة الواحدة يجب أن يشترك في: عملة أساسية واحدة (Base Currency)، وشجرة حسابات مالية واحدة (Chart of Accounts)، وتقويم عمل موحد (Calendar).
  4. مستوى الموقع (Site Level): المستوى التشغيلي الفعلي على الأرض (مثل فرع القاهرة وفرع الإسكندرية). البيانات هنا معزولة تماماً ولا يمكن لموقع الاطلاع على بيانات موقع آخر إلا بصلاحيات أمنية خاصة.

Launchpads & Installers

  • Product Launchpads: واجهة مركزية تطلق برامج التثبيت للمكونات والبرمجيات الوسيطة وتوضح لك الترتيب الصحيح لتثبيتها.

4. حزم وتجميع التطبيق: ملفات الـ EAR

Enterprise Archive

ما هو ملف الـ EAR؟ هو ملف مضغوط بامتداد .ear يعتمد على صيغة ZIP/ZLIB. هذا الملف يمثل حزمة التطبيق الكاملة في حاويات جافا.

💡 التشبيه البرمجي
في Laravel، عندما تقوم بحزم مجلد المشروع بالكامل داخل حاوية Docker Image لتوزيعها، فإن حاويات الجافا تفعل الشيء نفسه. ملف الـ EAR يحتوي بداخله على ملفات WAR (Web Archive — صفحات الويب والـ HTML والـ JSPs = الـ View) وملفات JAR (Java Archive — الـ Java Classes والبيزنس لوجيك = الـ Models/Controllers).

متى يجب بناء ملفات الـ EAR يدوياً؟

  1. إذا تم تأجيل Defer خطوة إعداد وتكوين الـ Java EE أثناء التثبيت أوتوماتيكياً.
  2. إذا قمت بتخصيص (Customize) أي ملفات أو إضافة ملفات جديدة إلى مجلدات الأبلكيشن سيرفر.

خطوات بناء وتوزيع الـ EAR عملياً

  1. افتح الـ Command Prompt على جهاز العمل.
  2. انتقل إلى مجلد التوزيع: cd C:\...\maximo\deployment
  3. قم بتشغيل سكريبت البناء: buildmaximoear.cmd و buildmxiehsear.cmd
  4. يستغرق الأمر بضع دقائق حتى تظهر جملة BUILD SUCCESSFUL (باستخدام أداة Apache Ant).
  5. لتوزيعه: تأكد من تشغيل الـ Deployment Manager والـ Node Agent، قم بإيقاف الـ MXServer أولاً، ثم ارفع ملف الـ EAR الجديد عبر Admin Console.

1. ما هي الـ Administrative Workstation؟

هي جهاز كمبيوتر يعمل بنظام Windows يتم تثبيت ملفات كود المحرك والمنتجات التابعة له عليه.

💡 التشبيه البرمجي
هي تماماً مثل لابتوب المطور أو سيرفر الـ CI/CD (مثل Jenkins أو GitHub Actions). في العمليات اليومية، لا يحتاج السيرفر أن يكون هذا الجهاز مفتوحاً. ولكن أهميته تكمن في أنه المكان الوحيد المعتمد لبناء ملفات الـ EAR، وتثبيت الـ Patches، وترقية الإصدارات، وإضافة حزم اللغات. من الممارسات الشائعة إنشاؤه كـ Virtual Machine (VM) لتسهيل نقله وحفظه.

2. هيكل مجلدات النظام القياسي

System Directory Structure

يتم تنظيم الملفات في هيكل مطابق لملف EAR مفتوح Exploded EAR Structure تحت المجلد الأساسي C:\ibm\SMP\maximo:

  • applications كود الجافا والمكونات؛ مجلد maximo يحتوي على الـ Java Classes الخاصة بكائنات العمل (MBOs)، ومجلد mxiehs يحتوي على الصور والـ Help.
  • deployment ملفات الـ EAR والـ WAR بعد بنائها، والسكريبتات المسؤولة عن تجميعها.
  • properties ملفات الإعداد النصية الحساسة مثل maximo.properties و doclink.properties.
  • reports ملفات تصاميم تقارير نظام BIRT وقوالبها.
  • resources ملفات الرسائل (Messages) وملفات الـ XML الخاصة بواجهات التطبيقات.
  • tools الأدوات والسكربتات التنفيذية لإعداد الداتا بيز مثل maxinst.bat و ConfigDB.bat.
🚫 تحذير كتابي صارم
يمنع تماماً تغيير هذا الهيكل التنظيمي للمجلدات أو زيادة عمق المجلدات (Add depth) لأن ذلك يتسبب فوراً في فشل البرامج والسكريبتات تلقائياً. كما يجب كتابة كافة أسماء المجلدات والملفات بالأحرف الصغيرة (lowercase) لتجنب مشاكل حساسية الأحرف عند النشر على خوادم UNIX.

1. ما هي خصائص النظام (System Properties)؟

هي عبارة عن متغيرات مخزنة على هيئة مفتاح وقيمة Key-Value Pairs يستخدمها النظام على مستوى السيرفر لضبط سلوك الأبلكيشن. يتم إدارتها عبر تطبيق System Properties Application.

أ. الخاصية العالمية (Global Property)

خاصية تنطبق على مستوى السيستم ككل System-wide Level. تؤثر على جميع خوادم التطبيقات التي تتصل بنفس قاعدة البيانات المشتركة، حتى لو كان النظام في بيئة مجمعة Clustered Installation.

مثال: الخاصية mxe.logging.rootfolder التي تحدد مسار مجلد حفظ ملفات اللوج.

الخصائص البرمجية:

  • Online Changes Allowed: تحدد هل يسمح بتعديل القيمة مباشرة من الشاشة. بعض الخصائص الحساسة مثل mxe.db.user لا تسمح بذلك.
  • Live Refresh: تطبيق القيمة الجديدة فوراً في الميموري دون إعادة تشغيل السيرفر.
  • Encryption: تشفير قيم المتغيرات في قاعدة البيانات تلقائياً لحماية كلمات المرور.
  • Security Level: ثلاثة مستويات وصول (Public, Secure, Private).

ب. خاصية المثيل/الخادم الفردي (Instance Property)

خاصية يتم ربطها وتخصيصها لخادم أبلكيشن معين بالاسم دوناً عن باقي الخوادم في الـ Cluster.

📌 القاعدة الذهبية للمفاضلة
إذا تم تحديد قيمة للمتغير على المستوى العام (Global) وتحديد قيمة أخرى لنفس المتغير على مستوى خادم معين (Instance)، فإن قيمة الـ Instance لها الأولوية القصوى وتعمل على عمل Override للقيمة العامة.
مثال عملي: إذا كان لدينا كلوستر يحتوي على 3 سيرفرات، ونريد تعطيل BBCron على السيرفر الثالث فقط MXServer3، نقوم بربط المتغير mxe.crontask.donotrun بالسيرفر MXServer3 ونعطيه قيمة BBCron. النتيجة: سيتوقف الـ Cron Task عن العمل في السيرفر الثالث فقط.

2. المفاضلة بين قيم الملف وقيم الشاشة

File vs Application

📌 القاعدة
إذا قمت بكتابة قيمة لمتغير داخل ملف الخصائص النصي وفي نفس الوقت قمت بكتابة قيمة أخرى في شاشة System Properties، فإن القيمة المكتوبة داخل الملف النصي لها الأولوية وتفوز تلقائياً. يتم تفعيل هذه الخاصية عند تفعيل خيار File Override.

الفائدة في بيئة التطوير: تتيح لعدة مطورين شغالين على نفس قاعدة البيانات المشتركة أن يقوم كل مطور بتشغيل الأبلكيشن سيرفر الخاص به بقيم متغيرات مختلفة عن طريق وضع ملف maximo.properties مخصص على جهازه المحلي.

Core Definitions

📍

3. أنواع المجموعات (Types of Sets)

🔩
مجموعة الأصناف (Item Set)

🏢
مجموعة الشركات (Company Set)
لتوحيد سجلات الموردين والمصنعين والجهات الخارجية Company Master. تضمن أن جميع المؤسسات تستخدم أسماء وأكواد موحدة للموردين، مما يسمح بعمليات شراء مركزية Centralized Purchasing.

المستوى المعرف الوصف أمثلة

3. معمارية مكونات الـ GL Account

GL Account Configuration

القيود والأحجام

  • حتى 20 مكوناً أو مقطعاً منفصلاً Up to 20 Components/Segments.
  • طول إجمالي يصل إلى 254 حرفاً أو رقماً بدون احتساب الفواصل.
  • GL Order: يحدد ترتيب ظهور المقاطع من اليسار لليمين؛ الرقم 0 = المقطع الأعلى في الهرمية.
مثال قياسي:
المقطع 1 = مركز التكلفة (Cost Center) → المقطع 2 = النشاط (Activity) → المقطع 3 = المورد (Resource) → المقطع 4 = العنصر (Element).
🚫 قانون المكونات الاختيارية

الـ System هو معرف هيكلي يربط ويجمع مجموعة من مواقع التشغيل في علاقات شبكية أو هرمية لغرض التقارير وتتبع الأثر Roll-up Costs.

🌲

علاقة شجرية تعتمد على نظام الـ Parent-Child. الفرع الرئيسي في القمة، ويتفرع منه المباني، ثم الأدوار، ثم الغرف.
🕸️

3. قوانين النظام الأساسي الصارمة

Primary System Rules

  • النظام الأساسي الإلزامي: يشترط المحرك أن يمتلك كل موقع Site نظاماً أساسياً واحداً فقط بالاسم Exactly One Primary System.
  • طبيعة النظام الأساسي: يجب أن يكون هرمياً (Hierarchical) حصراً، ولا يمكن أن يكون شبكياً.
  • جذر القمة الفريد: يشترط وجود موقع أعلى واحد فقط يمثل الجد الأكبر Single Top-Level Parent Location.
  • شرط هرمية الأصول: إذا كانت الأصول مرتبطة في هرمية معقدة Asset Hierarchy، يجب أن تكون مواقعها معرّفة داخل الـ Primary System.
  • تعدد الأنظمة: يمكن للموقع الواحد أن ينتمي لأكثر من نظام في نفس الوقت (مثلاً: هرمية المبنى + شبكة التكييف + الشبكة الكهربائية).
  • قانون التأسيس المتبادل: يشترط المحرك وجود موقع واحد على الأقل معرّف قبل السماح بإنشاء الـ System الأول.

1. البنية التأسيسية للتصنيفات

Classifications Terms

التصنيف Classification هو نظام هرمي مُنظم لتحديد وتوصيف وفهرسة السجلات داخل قاعدة البيانات.

  • Hierarchy (الهرمية) معمارية شجرية يكون فيها كل عنصر (ما عدا عنصر القمة) تابعاً لعنصر أعلى منه.
  • Parent & Child الـ Parent = العقدة الأعلى (مثل: Truck)، والـ Children = المستويات الأدنى (مثل: Heavy Truck, Light Truck).
  • Classification Path النص الهيكلي الكامل مفصولاً بشرطات مائلة (مثل: IT\Assets\Hardware\Computers).
  • Attribute (الخاصية) مواصفات فنية تُربط بالمسار التصنيفي (مثل: Memory، CPU، Manufacturer). قيمها نصوص أو أرقام.

2. معيار UNSPSC Framework

أفضل الممارسات الهندسية لبناء شجرة التصنيفات هي تبني معيار UNSPSC (كود الأمم المتحدة القياسي للمنتجات والخدمات).

  • نظام تكويد عالمي مفتوح يتكون من 5 مستويات هرمية صارمة.
  • يمنع تكرار إدخال المواد والأصول بشكل غير مقصود No Unintentional Duplicates.
  • يجعل أكواد ومواصفات الشركة متطابقة مع كشوفات الموردين الخارجيين، مما يسهل الـ Data Reconciliation.

3. المواصفات الفنية وتطبيقاتها المرنة

Specification Templates

التطبيقات المدعومة: الأصول والمواقع والمواد، التذاكر بأنواعها (Incidents, Problems, Service Requests)، أوامر الشغل والأنشطة والعقود والتغييرات.

💡 Classification Search Window

  • مستوى التصنيف العام: الشجرة تُحفظ على مستوى النظام بالكامل System Level لتكون متاحة للجميع.
  • التخصيص المشروط: يمكن ربط عقدة بمؤسسة أو موقع محدد (قيمها Null افتراضياً لتصبح عامة).

1. الـ Authentication مقابل الـ Authorization

🔑
Authentication (المصادقة)
عملية التحقق من هوية المستخدم والتأكد من أنه الشخص الذي يدعي أنه هو، عبر الـ User ID والـ Password.
التشبيه البرمجي: هي عملية الـ Auth::attempt() التي تتحقق من صحة الحساب.
🛡️
Authorization (التفويض)
عملية تحديد ما يُسمح للمستخدم بفعله بعد نجاح مصادقته. منح أو منع الوصول للموديولات، الشاشات، الأزرار.
التشبيه البرمجي: هي الـ Roles & Permissions والـ Middleware.

2. آلية الـ Authentication: Native مقابل LDAP

أ. الأمان المحلي (Native Authentication)

يقوم محرك TPAE نفسه بإدارة الحسابات بالكامل. يتم التحقق من الـ User ID والكلمات السرية بمقارنتها مع جداول قاعدة البيانات المحلية MAXUSER.

  • إنشاء وإدارة الحسابات والمجموعات من داخل تطبيقات الأمان في الواجهة.
  • إدارة كلمات المرور وقوانين تعقيدها بالكامل عبر النظام.
  • يدعم ميزة التسجيل الذاتي (Self-Registration).
  • نوع الدخول: Form-based Login.

ب. أمان خادم التطبيقات (LDAP Authentication)

يتنازل المحرك عن إدارة الحسابات، ويفوض خادم التطبيقات (WebSphere) ليتصل بـ LDAP Server مثل Active Directory.

  • إنشاء الحسابات والمجموعات وإدارة الباسوردات داخل الـ LDAP حصرياً.
  • يعتمد على Basic Authentication.
  • يدعم ميزة Single Sign-On (SSO) عبر آلية LTPA (Lightweight Third Party Authentication).

3. التحكم في إدارة البيانات عند استخدام LDAP

📌 الحل البرمجي
لفصل آلية المصادقة عن إدارة الحسابات، يتم تعديل خاصية النظام:
UPDATE MAXPROPVALUE SET PROPVALUE = '0' WHERE PROPNAME = 'mxe.LDAPUserMgmt';

عندما تأخذ الخاصية mxe.LDAPUserMgmt القيمة 0، تُفتح صلاحيات إنشاء المستخدمين وربط المجموعات من داخل تطبيقات المحرك مع بقاء التحقق من الباسورد تابعاً للـ LDAP.

1. التعريفات الأمنية الخمسة

Core Definitions

👤
Person (الشخص)
سجل يمثل فرداً حقيقياً (الاسم، العنوان، الإيميل). يمكن أن يظهر كجهة اتصال أو مستلم شحنة دون صلاحية دخول.
👥
Person Group
قائمة تجمع عدة أشخاص يتشاركون نفس المسؤوليات الوظيفية (مثل فريق صيانة الميكانيكا).
🔐
User (المستخدم)
الحساب الأمني الفعلي الذي يمتلك User ID ويقوم بتسجيل الدخول وإنشاء وتعديل السجلات.
⚒️
Labor (العمالة)
سجل لموظف أو مقاول داخلي يُسمح بتتبع ساعات عمله وتكلفته المالية داخل أوامر الشغل.
🛡️
Security Group
حزمة من الصلاحيات والامتيازات تُمنح لمجموعة من المستخدمين ذوي نفس الواجبات الوظيفية.

2. قوانين العلاقات والقيود

Relational Constraints

⚠️ التأسيس الإلزامي (The Core Rule)
يجب إنشاء سجل الـ Person أولاً قبل إنشاء User أو Labor. كل مستخدم أو عامل يجب ربطه بسجل شخص معرّف مسبقاً.
  • User ↔ Person: علاقة 1-to-1 حصرية. لا يمكن لشخص واحد امتلاك أكثر من حساب مستخدم.
  • Labor ↔ Person: داخل نفس المؤسسة Organization = علاقة 1-to-1. لكن يسمح بربط الشخص بأكثر من سجل عمالة في مؤسسات مختلفة.
  • الدمج: يمكن لسجل الشخص الواحد أن يتصل بسجل مستخدم وسجل عمالة في نفس الوقت.

3. المجموعات الأمنية الأربعة الافتراضية

Initial Security Groups

المجموعة الوظيفة
MAXADMIN الـ Super-User / Root — صلاحيات كاملة ومطلقة لعمل أي شيء.
DEFLTREG مجموعة افتراضية للمستخدمين الجدد. صلاحيات محدودة (تغيير الباسورد فقط).

طرق التعيين: عبر Set Security Profile داخل تطبيق المستخدمين، أو عبر My Profile في شريط الأدوات.

5. ميكانيكية الـ Access Control والمجموعات المتداخلة

Virtual Profile

تتم إدارة الصلاحيات على مستوى المجموعة Group Level وليس المستخدم الفردي. ينقسم نوع المجموعات إلى فئتين:

🔒
المجموعات المستقلة (Independent)
صلاحياتها معزولة تماماً ولا تندمج مع المجموعات الأخرى. يتم فحص صلاحيات كل تطبيق لكل موقع بشكل منفصل.
🔓
المجموعات غير المستقلة (Non-Independent)
مصممة للدمج والتراكم. المحرك يقوم بعمل Union / OR Operation ويمنح المستخدم أعلى مستوى وصول متاح Highest Access Wins.
مثال عملي: الموظف أحمد ينتمي للمجموعة (أ) = Read فقط، والمجموعة (ب) غير المستقلة = Save & Delete لنفس التطبيق. النتيجة: الـ Virtual Profile = Read + Save + Delete (أعلى مستوى).

أنواع الصلاحيات: Read، Insert، Save، Delete، بالإضافة لـ Signature Options (SigOptions) للتحكم في الأزرار والـ Actions.

1. تطبيقات الـ Conditional Security

يتيح تطبيق Conditional Expression Manager إنشاء وإدارة مكتبة مركزية من الشروط البرمجية المخزنة في النظام. يمكن استخدام هذه الشروط في ثلاثة محاور أمنية وتصميمية:

🔑
الوصول الشرطي للخيارات (Conditional Option Access)
ربط الـ SigOptions أو أزرار الـ Select Action بشرط أمني.
مثال: السماح للمجموعة الأمنية بالضغط على زر "اعتماد طلب الشراء" بشرط أن تكون تكلفة الطلب أقل من 10,000 دولار.
🚫
قيود البيانات (Data Restrictions)
فرض قيود أمنية على الكائنات أو الأعمدة.
مثال: جعل حقل "الراتب" مخفياً أو قراءة فقط إذا كان الموظف يقع في قسم آخر.
🎨
الواجهات الشرطية (Conditional UI)
تغيير خصائص عناصر الشاشة ديناميكياً.
مثال: تحويل لون حقل "تاريخ الانتهاء" إلى اللون الأحمر أو جعله إجبارياً (Required) بمجرد تحول حالة أمر الشغل إلى 'In Progress'.

2. أنواع الشروط في المحرك (Types of Conditions)

عند إنشاء شرط جديد، يقوم النظام بتوليد كود تلقائي له (Condition ID) ويمكنك الاختيار بين نوعين حصريين للتنفيذ:

  • EXPRESSION: كتابة كود شرطي يعتمد على قواعد الـ SQL syntax (مثل: :status = 'APPR').
  • CLASS: كتابة اسم ملف كلاس جافا مخصص (Java Class File) تم برمجته ورفعه على السيرفر لتنفيذ لوجيك معقد جداً يصعب صياغته بـ SQL.
⚠️ قاعدة التتبع الحرج (Reference Count)
يحتوي التطبيق على حقل اسمه Reference Count. هذا الحقل عبارة عن عداد برمجائي يزيد تلقائياً بمقدار 1 في كل مرة يتم فيها استخدام أو ربط هذا الشرط بتطبيق أو مجموعة أمنية أو عنصر واجهة. ويمنع النظام تماماً حذف أي شرط من المكتبة إذا كانت قيمة الـ Reference Count أكبر من صفر لضمان سلامة الروابط.

3. أداة الـ Conditional Expression Builder

لتسهيل كتابة الشروط دون الوقوع في أخطاء الـ Syntax، يوفر المحرك أداة Conditional Expression Builder.

⚙️ القاعدة البرمجية لعمل الأداة
يبدأ بناء أي تعبير شرطي باختيار كائن العمل الأساسي المستهدف (Object). وبمجرد تحديد الـ Object (مثل WORKORDER)، يقوم المحرك تلقائياً بعمل عملية فلترة ذكية لقائمة الخصائص (Attributes) ليعرض لك فقط الأعمدة التابعة لهذا الجدول حصرياً لتختار منها وتبني شرط الـ WHERE الخاص بك بسلاسة.

1. المكونات الأمنية لخادم التطبيقات (WebSphere Architecture)

عند تفعيل أمان الحاوية (Container Security)، يعتمد حل إدارة الخدمات بالكامل على مكونين رئيسيين داخل WebSphere لإتمام عملية الـ Authentication:

🗂️
Virtual Member Manager (VMM)
هو المكون المسؤول عن ربط وتجميع مستودعات الحسابات المختلفة (مثل عدة خوادم LDAP أو قواعد بيانات مختلفة) ودمجهم برمجياً ليظهروا للأبلكيشن كمستودع افتراضي واحد وموحد (Single Virtual Repository). يوفر الـ VMM واجهة برمجية مستقلة (Repository-independent API) تحمي تطبيق TPAE من تفاصيل وتغييرات نوع الـ LDAP؛ فالمحرك يكلم الـ VMM دائماً، والـ VMM يتكفل بالترجمة للخادم الخارجي.
🛡️
Secure Token Service (STS)
المكون المسؤول عن توليد وإصدار والتحقق من الـ Tokens والشهادات الأمنية لتمرير هوية المستخدم المصادق عليه بين تطبيقات Java EE المختلفة.

2. ميكانيكية الـ Synchronization (مزامنة البيانات)

⚠️ المشكلة المعمارية
يقوم الـ LDAP بإدارة والتحقق من هوية المستخدم وباسورده (Authentication). ولكن محرك TPAE مجبر على الاحتفاظ ببيانات هؤلاء المستخدمين والمجموعات داخل جداول قاعدة بياناته المحلية (مثل MAXUSER و MAXGROUP) لكي يتمكن من ربطهم بالصلاحيات الدقيقة وأوامر الشغل والأصول (Authorization).
✅ الحل التلقائي (One-Way Sync)
يتم تفعيل آلية الـ Synchronization لنقل الداتا. وهي حركة بيانات تتم في اتجاه واحد فقط (One-Way Direction) من خادم الـ LDAP إلى قاعدة بيانات المحرك. وتتم عبر جدولة وإطلاق كروت الـ Cron Tasks الجاهزة في النظام مثل كرت LDAPSYNC أو كرت VMMSYNC.

⚠️ قوانين وقواعد المزامنة الصارمة للـ Assessment:

يفرض الكتاب مجموعة من القواعد والتحذيرات السلوكية لآلية المزامنة والتي تختبر بدقة مدى فهمك للمعمارية:

  1. حذف الحسابات: إن حذف حساب مستخدم أو مجموعة أمنية بالكامل من خادم الـ LDAP لا يؤدي نهائياً لحذفهم من قاعدة بيانات المحرك تلقائياً.
  2. تعطيل الحسابات (The Lock Trap): إذا قمت بقفل أو تعطيل حساب موظف داخل الـ LDAP، فإن حسابه لن يتم تعطيله أو تصفيره داخل جداول المحرك أوتوماتيكياً؛ يجب على مدير النظام الدخول يدوياً لتطبيق المستخدمين وإلغاء تفعيل الحساب أمنياً لحماية الداتا.
  3. تغيير المسميات: إعادة تسمية (Rename) مستخدم أو مجموعة في الـ LDAP لا يعكس الاسم الجديد في جداول النظام المحلية.
  4. المطابقة الافتراضية وخريطة الحقول: يقوم كرت الـ Cron Task بالمزامنة الافتراضية بناءً على خاصية الاسم الكامل للـ LDAP والمعروفة بـ (cn / Common Name). ويمكن للمطور تعديل هذا السلوك وتغيير الخرائط البرمجية عن طريق تعديل بارامترات المتغيرات UserMapping و GroupMapping القابعة داخل إعدادات الـ VMMSYNC cron task لتوجيه الحقول للأعمدة المظبوطة في الجداول. ويشترط لتشغيل الـ Cron Task تزويدها ببيانات حساب الـ wasadmin لتتمكن من استدعاء واجهات الـ VMM بنجاح.

1. واجهة سطر الأوامر (Command-Line Interface / CLI)

تعتبر الواجهة السريعة لإدخال الحسابات بكميات ضخمة (Bulk Import) إلى سجل الـ LDAP. وتعتمد وظيفياً على أداة القياس ldapadd.

وتأخذ الأداة ملفات نصية مهيكلة تسمى ملفات الـ LDIF (LDAP Data Interchange Format) وتنقاد بالـ Parameters التالية:

  • -h لتحديد الـ IP أو الهوست لخادم الـ LDAP.
  • -p لتحديد منفذ الاتصال (افتراضياً 389 للاتصال العادي).
  • -D لتحديد حساب المدير المسؤول عن الحقن (الموثق بالـ Distinguished Name مثل cn=root).
  • -w لكتابة كلمة المرور.
  • -f لتحديد مسار ملف الـ .ldif الذي يحتوي على الداتا.
مثال الحذف يدوياً:
لحذف مستخدم عبر ملف ليدف، يجب صياغة حقل التغيير النوعي changetype: delete داخل السكريبت ليفهم خادم الأدلة الأمر.

1. صد هجمات التخمين والـ DOS Attacks

يمتلك المحرك آلية ذكية تراقب قنوات الاتصال والـ Traffic بشكل تلقائي. إذا استشعرت المنصة وجود نشاط مشبوه (Suspicious Activity)، يقوم النظام بإغلاق وقفل وعمل حظر (Block) لعنوان الـ IP الذي تنطلق منه الهجمة فوراً في الخلفية.

🚨 مثيرات الحظر التلقائي
يتم تفعيل ميزة الحظر إذا تخطى المستخدم عدد المحاولات الفاشلة المسموح بها في ثلاثة بوابات: (محاولات تسجيل الدخول الفاشلة الـ Login، محاولات استرجاع وتخمين كلمة المرور Forgotten Password، أو محاولات التسجيل الذاتي المتكررة من نفس الجهاز).

2. الخصائص البرمجية وجدول الـ LOGINBLOCK

تدار هذه المنظومة الدفاعية بالكامل عبر مجموعة من خصائص النظام الحساسة (System Properties) وجدول تتبع في قاعدة البيانات:

  • mxe.sec.IPblock متغير بوليان (Boolean) يأخذ القيمة 1 لتفعيل محرك حظر الـ IP أو 0 لتعطيله.
  • mxe.sec.IPblock.num يحدد سقف عدد المحاولات الفاشلة المسموح بها للمستخدم قبل إطلاق قرار الحظر.
  • mxe.sec.IPblock.sec يحدد النافذة الزمنية (الوقت بالثواني) التي يتم مراقبة المحاولات الفاشلة خلالها.
    مثال مدمج: إذا تم ضبط المحاولات بـ 5 والوقت بـ 60؛ وقام هاكر بعمل 6 محاولات فاشلة خلال 40 ثانية، يستشعر المحرك تخطي السقف أوتوماتيكياً ويقفل القناة.
  • mxe.sec.IPblock.MatchBoth خاصية أمان صارمة؛ في حال تفعيلها، لا يكتفي النظام بفحص الـ IP الصافي، بل يمنع ويعمل Block للـ IP فقط إذا تطابق كلاً من عنوان الـ IP واسم المضيف (Host Name) الخاص بجهاز العميل القادم مع السجلات المجهولة.
  • mxe.sec.addusers.maxsets يحدد الحد الأقصى لمحاولات التسجيل الذاتي المتزامنة لمنع هجمات إغراق السيرفر بالـ Garbage Accounts.
🗄️ جدول الـ LOGINBLOCK وآلية فك الحظر
جدول الـ LOGINBLOCK: الجدول المخصص في قاعدة البيانات الذي يقوم المحرك بكتابة وحقن الـ IPs المحظورة وتاريخ حظرها داخله تلقائياً.

آلية الإدارة وفك الحظر: لا يتم فك الحظر عن الأجهزة أوتوماتيكياً بمرور الوقت؛ بل يجب على مدير النظام (Administrator) الدخول يدوياً وتطبيق إجراء مخصص اسمه Manage Blocked IP Addresses الملحق بتطبيق المستخدمين (Users Application) لمراجعة جدول الحظر ومسح الـ IP لتنظيف جدول الـ LOGINBLOCK والسماح للجهاز بالاتصال مجدداً.

معمارية قاعدة البيانات (Database Architecture)

1. طبقة التجريد البرمجية لقواعد البيانات

تعمل تطبيقات Java EE (مثل محرك TPAE) فوق طبقة تجريد (Layer of Abstraction) تفصل بالكامل بين الكود البرمجي وبين طريقة التخزين الفيزيائية في قاعدة البيانات.

✅ قواعد البيانات المدعومة
يدعم المحرك بشكل قياسي ثلاث قواعد بيانات عالمية كمخزن دائم للبيانات (Persistent Data Store): IBM DB2, Oracle، و Microsoft SQL Server. يتم التعامل مع قواعد البيانات هذه عبر الـ Java EE API القياسي والـ Drivers المعتمدة (مثل JDBC).

كونه Database-Agnostic؛ عند كتابة كود أو إضافة ميزات، لا تشغل بالك بنوع الداتابيز المكتوبة، فالمحرك يتكفل بترجمة الاستعلامات أوتوماتيكياً للـ Dialect الصحيح.

2. كائنات العمل البرمجية (System Business Objects)

في بيئة TPAE، لا يتصل المطور أو التطبيق بالجداول مباشرة، بل يتم التعامل بالكامل على مستوى كائنات العمل (Business Objects).

في الخلفية، يقوم كائن العمل (والذي يمثل الـ Model أو الـ MBO) بسحب الصفوف والأعمدة من جداول قاعدة البيانات الملموسة ويقدمها للمستخدم والواجهات على هيئة خصائص (Attributes).

📌 تعدد الكائنات (Many-to-One Table)
يسمح النظام بتعريف أكثر من كائن بيزنس مستقل على جدول قاعدة بيانات واحد. كل كائن يمتلك بيزنس لوجيك وقوانين عمل (Business Rules) مختلفة تماماً، لكن داتا الكائنين تعيش وتخزن في نفس الجدول.

مثال عملي من الكتاب: كائنات العمل Incident (بلاغ عطل)، Problem (مشكلة)، و Ticket (تذكرة عمومية) هي ثلاثة كائنات عمل منفصلة في اللوجيك، ولكنها جميعاً تقرأ وتخزن في جدول ملموس واحد في الداتا بيز وهو جدول TICKET.

بفضل طبقة الـ MBOs، يتم إخفاء جداول النظام الداخلية المعقدة عن واجهة المستخدم لحمايتها، مع السماح للمسؤول فقط باستعراضها. الجداول الداخلية المدمجة في قاموس البيانات (Internal Data Dictionary Objects) للقراءة فقط ولا يمكن حذفها أو تعديلها.

3. خصائص الكائنات وحمايتها

الخاصية البرمجية (Attribute) هي المواصفة الفنية التي تحدد خاصية معينة لكائن البيزنس (تمثل اسم العمود وقيمته ونوعه).

🚨 قوانين الحذف والـ Indexes
يتيح النظام حذف الـ Attributes التي قام المستخدم بتعريفها (User-Defined Attributes). ولكن يفرض المحرك تحذيراً صارماً قبل الحذف: يجب على مهندس قاعدة البيانات الذهاب لتبويب الفهارس (Indexes Tab) أولاً للتحقق من أن هذا العمود مشمول داخل فوت برنت أو فهرس داتابيز؛ إذا كان مشمولاً في Index يجب إسقاط الفهرس أولاً قبل حذف الـ Attribute منعاً لضرب قاعدة البيانات.

4. النواة المركزية: قاموس بيانات النظام (The Data Dictionary)

قاموس البيانات (Data Dictionary) هو عبارة عن مجموعة من الجداول المركزية الحساسة التي تخزن الـ Metadata (البيانات التي توصف هيكل داتا السيستم وعلاقاتها). كافة مكونات الأبلكيشن سيرفر تعتمد عليه لترجمة الـ Java Objects إلى جداول علاقية حقيقية في الـ RDBMS.

اسم جدول النظام الوظيفة البرمجية والمعمارية
MAXOBJECT القائمة الرئيسية لكافة كائنات النظام؛ يربط الكائن البرمجي بالجدول (Table) أو الـ View المقابل له.
MAXTABLE يسرد ويخزن أسماء كافة الجداول الحقيقية الملموسة في قاعدة البيانات الفيزيائية.
MAXVIEW يسرد كافة الاستعلامات أو الرؤى المركبة والـ Views المعرفة في النظام.
MAXATTRIBUTE دليل الأعمدة والخصائص؛ يسجل مواصفات كل حقل لكل Object (نوعه، حجمه، هل هو مطلوب؟).
MAXVIEWCOLUMN تتبع ورسم تفصيلي لكافة الأعمدة التي يتشكل منها الـ View.
MAXRELATIONSHIP مخزن العلاقات؛ يحفظ أكواد الـ SQL Joins وشروط الربط بين كائنات الآباء والأبناء.
MAXSEQUENCE يسرد مولدات الأرقام التتابعية القياسية (Sequences) التي يعتمد عليها DB2 أو Oracle لإنشاء الـ Primary IDs.
MAXSYSINDEXES يسجل كافة الفهارس والـ Indexes المعتمدة رسمياً في دكشنري النظام.

5. أنواع ارتباطات الكائنات بجداول قاعدة البيانات (Object Associations)

يقسم دكشنري النظام الارتباطات إلى نوعين:

  • كائنات مرتبطة بجداول (Table Associations): إما كائن بيزنس واحد يشير لجدول واحد (1-to-1) أو عدة كائنات تشير لنفس الجدول (Many-to-1).
  • كائنات مرتبطة بـ Views: يتيح لكائن البيزنس أن يمتد ويجمع داتا من عدة جداول في نفس الوقت.
⚠️ شرط معماري صارم للـ Views
يشترط المحرك لعمل هذا الربط أن تشترك كافة الجداول المكونة للـ View في نفس مفتاح الحقل الأساسي (Same Primary Key)، وأن تجمعهم علاقة One-to-One صروحة. كما يمكن لكائنات البيزنس أن ترث وتعمل امتداد من بعضها البرمجي (Extend each other).

6. تطبيق تهيئة قاعدة البيانات (Database Configuration Tool)

هو الأداة الرسومية المعتمدة لإدارة الـ Schema وتعديل قاموس البيانات. من خلاله يتم تنفيذ المهام التالية:

  • تعديل تعاريف ومواصفات الأعمدة وحقن النطاقات وقوائم القيم (Domains) بها.
  • إنشاء كائنات مستخدم جديدة (User-Defined Objects) وأعمدة مخصصة وفهارس و Views.
  • تحديث إحصائيات الفهارس لتحسين سرعة الاستعلامات.
  • ضبط وإعداد كروت التوقيع الإلكتروني والتدقيق الرقمي (E-Signature & E-Audit).
🚫 تحذير أمني قاتل
نظراً لأن التغييرات في هذا التطبيق قد تتسبب في إسقاط أو مسح بيانات كاملة من السيرفر، يشترط الكتاب أخذ نسخة احتياطية كاملة (Full Database Backup) قبل لمس أي إعداد.

7. ماذا يحدث خلف الكواليس أثناء تعديل الهيكل؟ (Behind the Scenes)

عندما تقوم بتعديل هيكل Object أو جدول (مثل تغيير حجم عمود أو نوعه) والضغط على "تطبيق التكوين"، يقوم المحرك بسلسلة من العمليات الأوتوماتيكية الحذرة لحماية داتا الشركة:

  1. يقوم المحرك إعادة تسمية الجدول الأصلي (Rename) ليتحول إلى جدول احتياطي مؤقت في قاعدة البيانات (Backup Table).
  2. يقوم أوتوماتيكياً بإنشاء جدول جديد تماماً (New Table) يحمل الهيكل والتعديلات الجديدة التي صممتها.
  3. يقوم بعملية نسخ ونقل داتا (Copy Data) من الجدول الاحتياطي القديم ويصبها داخل الجدول الجديد المطور.
  4. يترك الجدول الاحتياطي القديم في الداتا بيز دون مسحه كأمان لك لكي تراجعه أو تحذفه يدوياً لاحقاً.
⚠️ قاعدة أوفررايد الحظر
إذا قمت بعمل تعديل هيكلي ثانٍ على نفس الجدول قبل مسح الجدول الاحتياطي الأول، فإن المحرك سيعتبر الجدول الحالي هو الاحتياطي الجديد، ويقوم بعمل Drop (مسح وإسقاط) للجدول الاحتياطي القديم أوتوماتيكياً ويحل محله، مما يعني ضياع النسخة الاحتياطية الأقدم.

8. أنواع البيانات الـ 14 الصارمة (Data Types)

يحدّد المنهج قائمة بالـ Data Types المسموح بها للأعمدة والخصائص، ويحذر من تغيير نوع البيانات للعمود بعد حشو السجلات بالداتا وبدء إنتاجها في العمل لأن ذلك يعد مخاطرة معمارية كبيرة. الأنواع هي:

ALN
نصوص ألفانيميرك (حروف كبيرة وصغيرة).
AMOUNT
تنسيق أرقام العملات والمبالغ (تتحكم به شاشة خيارات الحجم والكسور).
DATE
حقل تاريخ مالي. (معلومة حرجة جداً): يتم حساب وحفظ طول حقل التاريخ بالبايت (In Bytes) وليس بعدد الحروف الكاريكترز! ويشتق التنسيق من نظام الويندوز، والصيغة الوحيدة التي يقبلها محرك Oracle هي dd-mm-yy.
DATETIME
دمج التاريخ والوقت معاً في حقل واحد.
DECIMAL
أرقام عشرية دقيقة.
DURATION
تنسيق المدد الزمنية والوقت بالساعات.
FLOAT
أرقام عشرية طافية بدون حجم أو كسر مسبق التعيين.
INTEGER
أرقام صحيحة تقع في النطاق الصارم من -2147483648 وحتى +2147483647.
LOWER
نصوص تُجبر برمجياً على التحول للأحرف الصغيرة فقط.
SMALLINT
رقم صحيح صغير من -32768 وحتى +32767.
TIME
قيمة وقت وتأخذ افتراضياً تنسيق hh:mm:ss.
UPPER
نصوص تُجبر على التحول للأحرف الكبيرة فقط (مهمة جداً لأكواد الحقول المفتاحية).
YORN
حقل منطقي يقبل نعم أو لا (Yes/No) فقط، ويتم حقنه وكتابته في قاعدة البيانات الفيزيائية على هيئة رقم صريح 1 أو 0.

9. تبويبات تطبيق إعدادات قاعدة البيانات (Application Tabs)

  • List Tab لبحث بفلتر QBE عن كائنات قاموس البيانات.
  • Object Tab يستعرض الخصائص العامة وينقسم لـ ثلاثة أقسام هيكلية: (Details لتفاصيل الاسم والجدول، View لكود السيكول للرؤى، و Audit لربط وتفعيل جدول التدقيق الرقمي).
  • Attributes Tab يعرض الخصائص؛ وينقسم لـ قسمين: Details لربط النطاقات والـ Domains، وقسم Advanced لـ تفعيل ميزة التوقيع الإلكتروني E-signature للحقل.
  • Indexes Tab لاستعراض الفهارس؛ حيث يعرض جدول الفهارس المعتمدة وتحته جدول الـ Columns المكونة للفهرس المظلل.
    القاعدة: لا يمكن تعديل فهرس قائم؛ يجب عمل Drop (إسقاط) ثم إعادة إنشائه من جديد. ويقوم النظام بإنشاء Index تلقائي فوراً لأي حقل يتم تعليمه كـ Primary Key. ويحظر استخدام هذا التبويب نهائياً مع الكائنات الممتدة أو الموروثة (Inherited/Extended Objects).
  • Relationships Tab الواجهة الرسومية المخصصة لبناء أكواد الـ SQL Joins لربط كائنات الآباء بالأبناء (Parent-Child Objects) وتمرير الداتا.

10. الكلمات المحجوزة للنظام (Database Reserved Words)

المفهوم: تمتلك قواعد البيانات وأنظمة ANSI كلمات محجوزة سلفاً لبناء الاستعلامات (مثل SELECT, WHERE, TABLE).

💡 آلية التحقق
عند إنشاء عمود أو كائن جديد، يقوم المحرك بإطلاق عملية فحص ضد الداتابيز الملموسة؛ إذا أرجعت قاعدة البيانات خطأ في الـ Syntax، يستنتج المحرك تلقائياً أن المطور كتب كلمة محجوزة (Reserved Word) ويخرج رسالة حظر لمنع المشاكل.

11. الفرق المصيري بين "الحفظ" و"التطبيق" (Saving vs Viewing Changes)

  • عملية الحفظ (Saving): عند الضغط على زر "حفظ" بعد تعديل عمود، لا يتم تعديل قاعدة البيانات الفيزيائية نهائياً. يقوم المحرك فقط بحفظ التعديلات المقترحة على هيئة صفوف مؤقتة داخل جدول إعدادات ثانوي يسمى جدول الـ cfg table، وتتحول حالة السجل إلى حالة معلقة (Pending Status).
  • ميزة الـ Temporary Storage: تتيح لك هذه المعمارية مراجعة وإكمال التعديلات لاحقاً دون ضياعها، مع إمكانية استخدام خيار Discard Changes لإلغاء المقترحات كلياً (وهي عملية غير رجعية وتأخذ بعض الوقت).
⚠️ معلومة مهمة للـ Assessment
لا يسمح المحرك بعمل استعلامات سيكول أو البحث داخل حقل الـ Status الخاص بالتغييرات المعلقة.

12. طرق تطبيق التغييرات النهائية (Applying Configuration Changes)

لكي تنتقل الداتا من جدول الـ cfg المؤقت وتعدل الـ Real Production Tables، هناك طريقتان حصريتان لتشغيل الـ Configuration:

الطريقة آلية العمل والشروط التقنية الصارمة مسار ومكان ملف الـ Log
واجهة سطر الأوامر (Command Line) يتم تشغيل السكريبت التنفيذي configdb.bat من محطة العمل الإدارية. (شرط أمني إجباري): يجب إغلاق وعمل Shut Down كامل للـ Application Server كلياً قبل تشغيل الأمر يدوياً. يتم كتابة وحقن الـ Log داخل مجلد الأدوات الإدارية: /tools/maximo/log ويسمى الملف ConfigDB+.log.
من واجهة الأبلكيشن مباشرة (GUI) يتم تشغيل إجراء Apply Configuration Changes من قائمة خيارات النظام مباشرة دون قفل الأبلكيشن سيرفر كلياً. (الشرط الصارم): يمنع المحرك تشغيل الخيار إلا بعد فرض وتفعيل "وضع المسؤول" (Admin Mode) أولاً. يتم كتابة وحفظ ملف الـ Log داخل المجلد الرئيسي لخادم التطبيقات (Application Server's Home Directory).

13. تفكيك وضع المسؤول (Admin Mode)

المفهوم: هو حالة خاصة وخافرة يتم وضع الـ Application Server (MXServer) فيها للسماح بتعديل الـ Schema أونلاين دون إغلاق كامل للسيرفر.

القيود الصارمة المفعّلة أثناء الـ Admin Mode: بمجرد تشغيله، يتم تفعيل القيود الهيكلية التالية لحماية نقل الداتا:

  1. تعطيل ووقف كافة المهام المجدولة في الخلفية (All Cron Tasks are Suspended) تلقائياً.
  2. منع وحظر استقبال أي اتصالات ريموت قادمة من الـ Java Clients الخارجية.
  3. يتم بث وإرسال رسائل تحذيرية للموظفين الحاليين عبر بورتلت لوحة الإعلانات (Bulletin Board) تطلب منهم الخروج.
  4. يحدد المسؤول مهلة زمنية بالدقائق (مثلاً 5 دقائق)؛ بمجرد انتهائها، يقوم السيرفر بإنهاء وطرد كافة جلسات المستخدمين المعلقة قسرياً (Forcefully Terminated) ما عدا عدد محدد من حسابات المديرين المصرح لهم بالدخول لإتمام التكوين.

14. حذف الجداول الاحتياطية وإعادة التزامن

  • المشكلة: عند انتهاء الـ ConfigDB، تظل جداول الباك أب القديمة موجودة في قاعدة البيانات الفيزيائية.
  • القاعدة الصارمة للـ Command Line: يمنع المحرك المطور من تشغيل سكريبت الـ configdb.bat من سطر الأوامر مرة أخرى في المستقبل إلا بعد مسح وتنظيف جداول الباك أب القديمة تماماً أولاً.
  • طرق المسح: إما عبر إجراء Delete Backup Tables من شاشة النظام، أو تشغيل سكريبت أداة الحذف المخصصة dropbackup.bat القابعة في مسار مجلد الأدوات: \maximo\tools\maximo.
  • أداة الـ Refresh Index Tables: إجراء مخصص في القائمة؛ وظيفته أن يذهب لقاعدة البيانات الفيزيائية الملموسة ليفحص الفهارس التي صممها الـ DBA الخارجي يدوياً هناك أو التي صممها نظام MS SQL Server تلقائياً، ويقوم بسحب مواصفاتها وحقنها في دكشنري وميتا-داتا النظام ليحدث التزامن بين الطرفين.

15. خيارات الأداء وتعديل أحجام الكسور

  • تحديث الإحصائيات (Update Statistics): إجراء يعيد تنظيم وهيكلة الفهارس بناءً على حجم الداتا الحالي لتحسين سرعة محركات الاستعلام (Query Optimizers). مفيد جداً بعد عمليات الحقن الكثيف للبيانات أو المسح الشامل، ولا ضرر من جدولته برمجياً ليعمل يومياً.
    الشرط المعماري: يأخذ وقتاً طويلاً جداً، ويشترط خلو النظام تماماً من أي مستخدمين يقومون بإدخال أو تعديل السجلات أثناء تشغيله.
  • حجم الحقول الفنية والكسور (Field Length and Format): نافذة تتيح تعديل المواصفات الافتراضية الشاملة لكافة الأعمدة التي تحمل نوع البيانات AMOUNT (المبالغ المالية):
    • Amount Length: الحجم الأقصى الافتراضي هو 10 خانات، ويمنع النظام تقصير الحجم لو كانت هناك داتا مسجلة في الجداول تتخطى الحجم الجديد المقترح.
    • Amount Scale: عدد الخانات المخصصة بعد العلامة العشرية والكسور (الافتراضي 2). (قاعدة حسابية صارمة): إذا قررت زيادة حجم الكسر العشري بمقدار 1، إنت مجبر قانوناً بزيادة الطول الكلي للحقل Amount Length بمقدار 1 بالتوازي.
    • Integer Length (الافتراضي 12)، و Small Integer Length (الافتراضي 10).
📌 الفروق بين قواعد البيانات في حقل الـ AMOUNT
نظام MS SQL Server ذكي ويقوم بتطبيق تغييرات الـ Format فوراً في الجداول دون الحاجة لتشغيل الـ ConfigDB. بينما نظام Oracle يتطلب إلزامياً تشغيل الـ Database Configuration لكي يغير هيكل الجدول وتأخذ التغييرات مفعولها. ويتم ضبط الرموز التعبيرية للعملة وسلوكها من Regional Settings الخاصة بلوحة تحكم السيرفر.

النطاقات (Domains)
Domains • نطاقات صيانة وفلترة قيم الحقول
▼

1. الخصائص الفنية للنطاقات

الـ Domain (ويسمى أيضاً Value List) هو قائمة محددة ومسبقة التكوين من القيم الصالحة.

عند ربط الـ Domain بـ Attribute معين لكائن البيزنس، يتم قفل الحقل برمجياً ويمنع النظام تماماً قبول أو إدخال أي قيمة من قِبل المستخدم إلا إذا كانت متطابقة وموجودة داخل عناصر هذا الـ Domain.

⚠️ قاعدة العرض الحتمية (Does NOT change UI presentation)
ربط الـ Domain بالحقل لا يغير نهائياً شكل أو طريقة عرض الحقل على شاشة المستخدم ولا يحوله تلقائياً لقائمة منسدلة Drop-Down List؛ الـ Domain يحكم فقط "صلاحية البيانات وقبولها في الباك إند". لتغيير طريقة العرض يدوياً وإضافة زر الـ Lookup، يجب الذهاب لتطبيق Application Designer أو كتابة كود جافا لربط عنصر الواجهة بالـ Lookup المظبوط.

2. الأنواع الستة الصارمة للـ Domains

تتم إدارة وإنشاء كافة النطاقات عبر تطبيق موحد يسمى Domains Application في موديول تهيئة النظام. وتنقسم برمجياً إلى 6 أنواع حصرية:

ALN Domain (Alphanumeric)
يحتوي على قائمة استاتيكية صلبة من القيم النصية الثابتة المخزنة في النظام ومواصفاتها الوصفية.
NUMERIC Domain
يحتوي على قائمة ثابتة من القيم الرقمية (ويدعم الأنظمة العشرية والكسور)، ويتم ضبط الطول والـ Scale الخاص بها. ويمكن ربط عناصر النطاقات النصية والرقمية بمؤسسة معينة (ORGID) أو موقع محدد لفلترتها.
NUMERIC RANGE Domain
لا يعتمد على قيم فردية مفسرة؛ بل يحدد سقفاً ونطاقاً مفتوحاً عبر حدود رقمية دنيا وعليا (Minimum & Maximum Thresholds) لتقييد الإدخال.
TABLE Domain (النطاق الديناميكي)
لا يحمل قيمًا ثابتة؛ بل يوجه المحرك ليسحب ويقرأ قائمة القيم ديناميكياً من عمود محدد داخل جدول آخر في قاعدة البيانات في الوقت الفعلي.
CROSSOVER Domain
محرك نقل وتوريث البيانات. لا يظهر كقائمة خيارات للمستخدم نهائياً. وظيفته هي نسخ ونقل قيم تلقائياً من حقول جدول المصدر إلى حقول جدول الهدف بمجرد حدوث تزامُن أو اختيار سجل.
SYNONYM Domain
هي نطاقات مدمجة تأتي مشحونة مع كود المنصة الأصلي من IBM للحكم وتسيير المنطق البرمجي الأساسي للسيستم (مثل نطاقات حالات السجلات WOSTATUS أو POSTATUS).

3. تفصيل الـ TABLE Domain وآلية التكوين

يتطلب تحديد اسم الكائن المستهدف (Object) واسم العمود المراد القراءة منه (Column). ويحتوي على حقلين شروط حاسمين:

  1. List Where Clause: كود سيكول لتصفية وفلترة القيم التي تظهر للموظف في قائمة الخيارات (مثل: status = 'ACTIVE').
  2. Validation Where Clause: كود سيكول صارم يستخدمه الباك إند للتحقق من صحة القيمة المكتوبة يدوياً ومطابقتها للشروط لمنع التلاعب.

ويتم ربطه بـ Error Message Group و Error Message Key المعرفين في جداول الرسائل لإخراج رسالة خطأ مخصصة ومفهومة للمستخدم إذا فشلت عملية التحقق.

4. تفصيل الـ CROSSOVER Domain وخيارات التحكم الفيدرالية

للتشكيل البرمجي: تحدد الـ Source Object، والشرط، ثم ترسم جدول الربط: (العمود في المصدر ← العمود المقابل في الهدف).

  • Copy Even If Source Is Null في حال تفعيله، لو كانت خانة المصدر فارغة (NULL)، سيقوم المحرك بمسح الداتا في الهدف ويجعلها NULL أيضاً. ولو أُلغي، يحتفظ الهدف بقيمته القديمة دون تغيير.
  • Copy Only If Destination Is Null في حال تفعيله، يقوم المحرك بحقن الداتا وكتابتها في الهدف في حالة وحيدة فقط: إذا كان حقل الهدف فارغاً تماماً كصفحة بيضاء؛ أما لو كان الموظف قد كتب فيه داتا يدوياً، يمتنع المحرك عن الكتابة فوقها ويحمي داتا المستخدم.
🚨 تحذير المطورين الحرج
شاشة الـ Crossover Domains لا تقوم بعمل Validation أو فحص لكود الـ SQL المكتوب في خانات الشروط؛ لو كتبت اسم عمود خطأ أو Syntax مكسور، لن تمنعك الشاشة أثناء الحفظ، ولكن سيضرب الأبلكيشن فوراً في الـ Runtime أثناء معالجة الحركات.

5. قواعد وقوانين الـ SYNONYM الصارمة للـ Assessment:

  1. يمنع تماماً إنشاء أو إضافة نطاق SYNONYM جديد في النظام.
  2. يمنع تماماً تعديل أو إضافة أو مسح القيم الداخلية التابعة لـ IBM والمعروفة بـ الـ (Internal Value / MaxValue)؛ لأن كود الجافا في السيرفر مكتوب ليتعرف على هذه القيم المحددة خارجياً.
  3. المسموح به فقط: يتيح لك النظام إنشاء قيم خارجية مرادفة ومخصصة لشركتك تسمى الـ (External Value / Value) تظهر للموظفين على الشاشة، بشرط ربطها وتبعيتها لـ قيمة داخلية معينة.
    مثال ملموس: القيمة الداخلية الثابتة لأمر الشغل هي APPR (معتمد). يمكنك تفريع مرادفات خارجية منها تناسب فروع شركتك مثل C_APPR (معتمد من فرع القاهرة) و A_APPR (معتمد من فرع الإسكندرية). كود الجافا في السيرفر سيعامل الكودين كـ APPR قياسي، ولكن الموظفين يرون مسمياتهم الخاصة.
  4. قانون الافتراض الاستاتيكي: يسمح لك النظام بجعل عدة قيم خارجية تشير لنفس القيمة الداخلية، ولكن يشترط برمجياً أن يمتلك الجروب قيمة خارجية واحدة فقط معلّمة كـ "Default" (افتراضية) لكل قيمة داخلية.

E-Signature & E-Audit • التحقق الأمني المتقدم والتدقيق الفيدرالي
▼

1. وظيفة محركات الأمان الإضافية

يوفر المحرك طبقة أمان وتدقيق متقدمة جداً تتوافق مع القوانين الحكومية الصارمة (مثل معايير FDA 21 CFR Part 11 للمؤسسات الحيوية والطبية العسكرية) للتحكم الكامل في مراقبة وصيانة تتبع البيانات.

2. التوقيع الإلكتروني (Electronic Signatures / E-Signature)

عند تفعيل الـ E-Signature على أكشن معين (مثل اعتماد أمر شراء) أو عند تعديل عمود محدد في جدول، يقوم المحرك بوقف المعاملة برمجياً وفوراً، ويطلق نافذة منبثقة على شاشة المستخدم تسمى الـ (Electronic Signature Authentication Window).

✅ شروط العبور الحتمية
يُجبر المستخدم على:
  1. إعادة كتابة كلمة المرور الخاصة بحسابه شخصياً لتأكيد الهوية.
  2. كتابة وتحديد "سبب التعديل" أو البيزنس ريزون من قائمة معتمدة.
إذا فشل في كتابة الباسورد المظبوط، يتم إلغاء المعاملة ويسجل النظام محاولة فاشلة. يتم تفعيلها من مجلد الخصائص المتقدمة للـ Attributes أو موديول الـ SigOptions للأزرار والقوائم.

3. التدقيق الرقمي وجداول الظل (Electronic Audit Records / E-Audit)

عند تفعيل الـ E-Audit على كائن عمل (من تبويب Object في شاشة الداتابيز)، يتحول النظام البرمجي إلى كاشف رادار كامل لكافة حركات الجدول الملموس.

⚠️ جداول الظل المفتوحة (A_ Tables)
بمجرد تفعيل الخيار وتشغيل الـ ConfigDB، يقوم المحرك تلقائياً بإنشاء جدول ظل موازي في قاعدة البيانات الفيزيائية يحمل نفس اسم الجدول الأصلي مسبوقاً بحرف الـ A_ (اختصار لـ Audit).

مثال: لتفعيل التدقيق على جدول أوامر الشغل WORKORDER؛ يتم تلقائياً بناء جدول ظل اسمه A_WORKORDER.

4. ماذا يسجل جدول الـ Audit تلقائياً في الخلفية؟

عند قيام أي مستخدم بـ (إضافة INSERT، تعديل UPDATE، أو مسح DELETE) لصف في الجدول الأصلي، يقوم المحرك بـ:

  1. أخذ لقطة كاملة للبيانات القديمة قبل التعديل والبيانات الجديدة بعد التعديل وحقنها في الـ A_ Table.
  2. تسجيل كود نوع المعاملة التتبعية تلقائياً.
  3. تسجيل الـ User ID الصريح الذي قام بالعملية، مع ختم بصمة تاريخ ووقت المعاملة بالملي ثانية الحقيقية للسيرفر، مما ينتج عنه سجل مراجعة فيدرالي كامل مستحيل التلاعب به أو مسحه (Immutable Audit Trail).

تظهر هذه البيانات للمسؤولين لاحقاً عبر تقارير تتبع دورة حياة السجل والمسمية بـ Package Lifecycle Reports لضمان الحوكمة الكاملة.

▼

1. تعريف أمر الشغل وهيكل الجدول

Work Orders & Schema

أمر الشغل (Work Order): هو السجل المركزي الذي يصف حقيقة وبصمة العمل المراد تنفيذه (سواء كان صيانة وقائية، صيانة طارئة، أو إصلاحاً مبرمجاً). يحتوي على تفاصيل المشكلة، المعدة المستهدفة (Asset)، مكانها (Location)، الميزانية المالية المرتبطة بها (GL Account)، والموارد البشرية والمواد المطلوبة لإنجازه.

جدول الـ WORKORDER: في قاعدة البيانات الفيزيائية (DB2)، تصب كافة أوامر الشغل داخل جدول واحد مركزي عملاق وهو جدول WORKORDER.

2. التريكة المعمارية: فئات أمر الشغل

Work Order Classes — Single Table Inheritance

على الرغم من أن النظام يحتوي على تطبيقات مختلفة تماماً في الواجهة، إلا أن كافة هذه التطبيقات تخزن بياناتها بالكامل داخل نفس الجدول الملموس وهو جدول WORKORDER.

💡 كيف يفرق المحرك بينها برمجياً؟
يمتلك الجدول عموداً بنيوياً حاسماً يسمى عمود الـ WOCLASS. يأخذ هذا العمود قيمة نصية تحدد نوع الكائن وقواعد العمل (Business Rules) التي سيمر بها في الباك إند. هذا النمط يتوافق تماماً مع نمط تصميم Single Table Inheritance الشهير في البرمجة.
قيمة الـ WOCLASS التطبيق المسؤول في الواجهة طبيعة الاستخدام البرمجي
WORKORDER Work Order Tracking لأوامر الصيانة والتشغيل القياسية للأصول والمعدات.
ACTIVITY Activity Tracking للمهام الفرعية خفيفة الوزن أو الأنشطة المستقلة التي لا تحتاج لدورة حياة معقدة.
CHANGE Changes Application مخصص لإدارة التغييرات الحرجة في البنية التحتية للـ IT (متوافق مع معايير ITIL).
RELEASE Releases Application مخصص لإدارة عمليات إطلاق وتوزيع حزم البرمجيات أو المكونات الضخمة.

3. نوع العمل والأولويات الموروثة

Work Types & Inherited Priority

  • نوع العمل (Work Type): كود معرّف يحدد فئة أمر الشغل لغرض الفلترة وإخراج التقارير (مثل: CM لصيانة تصحيحية، PM لصيانة وقائية، EM لطوارئ). يتم ضبط إعداداته على مستوى المؤسسة (ORGID).
  • أولوية أمر الشغل (Work Order Priority): رقم يحدد مدى أهمية وحرارة تنفيذ العمل (عادة من 0 إلى 99).
✅ محرك حساب الأولوية التلقائي (Inherited Priority)
عندما يفتح المستخدم أمر شغل جديد ويربطه بـ Asset أو Location، يقوم المحرك برسم حسابه أوتوماتيكياً؛ حيث يقرأ قيمة الأولوية الخاصة بالـ Asset وقيمة الأولوية الخاصة بالـ Location، ويدمج بينهما بناءً على مصفوفة وزن الأولوية المعرفة في خيارات المؤسسة ليقترح الأولوية النهائية لأمر الشغل تلقائياً.

4. محرك التحكم في التدفق التلقائي

Flow Control Engine

المفهوم: ميزة بوليان يتم تفعيلها على مستوى أمر الشغل أو الخطة.

💡 آلية العمل المعمارية — Middleware Chain
عند تفعيل الـ Flow Control، يتحول المحرك إلى مدير صارم للتتابع؛ حيث يمنع الفنيين من تغيير حالة المهام الفرعية (Tasks) يدوياً. يقوم المحرك بقفل الحالات، وبمجرد قيام فني بإنهاء المهمة رقم 10 وتحويلها إلى COMP (مكتملة)، يستشعر المحرك ذلك في أجزاء من الثانية ويقوم تلقائياً بفتح وقفل وتغيير حالة المهمة رقم 20 التالية في السلسلة وتحويلها إلى APPR (معتمدة) لبدء العمل عليها.

5. دورة حياة الحالات والقيود المالية

Work Order Statuses — WOSTATUS Synonym

يمر أمر الشغل بدورة حياة فيدرالية صارمة تحكم قبول الحركات المالية والقيود في قاعدة البيانات عبر نطاق الـ Synonym المسمى WOSTATUS:

الحالة المسمى القيود والقواعد المالية
WAPPR Waiting for Approval الحالة الافتراضية (Draft)؛ لا يمكن صرف مواد أو حجز عمالة.
APPR Approved تم الاعتماد مالياً وتشغيلياً. تُفتح بوابات حجز المواد واستهلاك الساعات.
WSCH Waiting on Schedule معتمد ولكن ينتظر توفير العمالة أو جدولة موعد التنفيذ.
WMATL Waiting on Material العمل متوقف وينتظر وصول قطع غيار أو مواد من المورد الخارجي.
INPRG In Progress العمل قيد التنفيذ الفعلي على الأرض من قِبل الفنيين.
COMP Completed انتهى العمل المادي. يسمح بتسجيل ساعات متأخرة أو صرف مواد تكميلية، ولكن يمنع تعديل تفاصيل خطة العمل الأساسية.
CLOSE Closed الختم النهائي والتاريخي. يتم تجميد السجل وقفل القنوات المالية نهائياً.
CAN Canceled إلغاء العمل. يشترط عدم وجود أي تكاليف مسجلة مسبقاً.
🚨 أهم قاعدة مالية وأمنية في الـ Assessment
بمجرد تحول حالة أمر الشغل إلى CLOSE، يتم تجميد السجل بالكامل وقفل قنواته المالية نهائياً؛ يمنع المحرك تماماً قبول أي حركات مالية (No labor reporting, no material issues, no tool invoices) تُحمل على هذا الحساب لمنع حدوث فساد محاسبي في الميزانيات المقفلة.

1. مفهوم خطة العمل وعلاقتها الهيكلية

Job Plans — The Blueprint Pattern

خطة العمل (Job Plan): هي بمثابة الـ Blueprint أو الـ Template (القالب القياسي) الذي يصف تفاصيل كيفية تنفيذ مهمة معينة تتكرر باستمرار (مثل: خطة صيانة تكييف مركزي، أو خطة فحص سيرفر).

💡 التشبيه البرمجي
الـ Job Plan هي الـ Abstract Class أو الـ Factory Configuration المعرفة سلفاً في النظام. بينما أمر الشغل (Work Order) هو الـ Instantiated Object (الكائن الحي في الميموري والداتا بيز) الذي تم توليده بناءً على هذا القالب ليعمل في تاريخ محدد وموقع محدد.

مستويات الحفظ الفيدرالية: تتيح المعمارية مرونة عالية في تحديد نطاق عمل الـ Job Plan؛ حيث يمكن إنشاؤها على مستوى النظام ككل (System Level) لتكون متاحة للجميع، أو تقييدها على مستوى مؤسسة محددة (ORGID)، أو موقع تشغيلي فرعي محدد كلياً (SITEID).

3. المعمارية المتقدمة: خطط العمل المتداخلة والديناميكية

Nested & Dynamic Job Plans

  • خطط العمل المتداخلة (Nested Job Plans): ميزة هندسية تتيح ربط ودمج خطة عمل بالكامل لتصبح خطوة فرعية (Task) داخل خطة عمل أكبر. مفيدة جداً في عمليات الصيانة الكبرى المتراكمة (مثل خطة الصيانة الخمسية لمصنع تحتوي في بطنها على خطط الصيانة السنوية والشهرية كمكونات فرعية مدمجة).
✅ خطط العمل الديناميكية (Dynamic Job Plans)
تريكة برمجية عبقرية لحساب الموارد أوتوماتيكياً بناءً على حجم ومقدار الأصول المستهدفة بدلاً من الأرقام الثابتة.

مثال عملي: الشركة تريد فحص "أعمدة الإنارة" في طريق ما. فحص العمود الواحد يتطلب ساعة عمل ومادة واحدة. لو فتحنا أمر شغل لـ طريق فيه 10 أعمدة، تقوم الـ Dynamic Job Plan بقراءة الـ Multiplication Factor (عامل الضرب) أوتوماتيكياً، وتضرب الموارد في 10، ليحسب الباك إند تلقائياً أن المطلوب هو 10 ساعات عمل و10 مواد.

1. ما هو تطبيق الـ Application Designer؟

هو أداة تطوير مدمجة داخل النظام (WYSIWYG Visual Editor) تتيح للمطورين تعديل واجهات التطبيقات الحالية أو بناء تطبيقات جديدة كلياً دون الحاجة لكتابة كود واجهات تقليدي.

التأثير في قاعدة البيانات: عند تعديل تصميم الشاشة، لا يقوم النظام بتعديل ملفات فيزيائية؛ بل يعدّل كود الـ XML المخزن في جدول النظام الشهير MAXPRESENTATION.

2. مسار التعديل المتقدم: التصدير والاستيراد

Export / Import XML Workflow

  1. الـ Export: يفتح المطور التطبيق المستهدف داخل الـ Application Designer ويضغط على زر Export Application Definition. يقوم المحرك بتجميع كود الـ XML من جدول MAXPRESENTATION وتحميله كملف نصي على جهاز المطور (مثل ملف wotrack.xml).
  2. الـ Edit: يتم فتح ملف الـ XML على محرر أكواد خارجي (مثل VS Code) لتعديل الـ Tags والـ Properties يدويًا. وهي طريقة آمنة وسريعة لعمل الـ Bulk Changes أو نقل المكونات.
  3. الـ Import: يعود المطور للشاشة ويضغط على أيقونة Import Application Definition ويرفع ملف الـ XML المعدل. يقوم المحرك بعمل الـ Parse للـ Tags ويتحقق من صحة المكونات، ثم يعمل Overwrite للصف المقابل في جدول الداتا بيز.

هذه هي النقطة الجوهرية؛ كيف تتصل عناصر الواجهة بجداول قاعدة البيانات الفيزيائية؟

  • Attribute كتابة اسم العمود الصريح في الداتا بيز المراد ربطه بهذا الحقل (مثل: WONUM أو STATUS).
  • Data Source ID (اختياري) تحديد المعرف البرمجي للجدول أو الـ MboSet الحاضن؛ وإذا تُرك فارغاً، يرث الحقل تلقائياً الـ Main Object الخاص بالتطبيق.
✅ قانون الوراثة التلقائية للخصائص
بمجرد ربط الـ Control بالـ Attribute، يرث الحقل في الواجهة كافة القيود والمواصفات المعرفة له في الـ Data Dictionary (MAXATTRIBUTE) أوتوماتيكياً.

لو كان العمود معرّفاً في الداتا بيز بنوع ALN وحجم 10، ومربوطاً بـ Domain معين، فإن الـ Text Box في الواجهة سيلتزم بالحجم أوتوماتيكياً، ويرفض قبول حروف خارج الـ Domain، ويتحول لـ Read-Only لو كان العمود معرّفاً كذلك في الداتابيز، دون الحاجة لكتابة كود فحص إضافي في الفرونت إند.

1. خيارات التوقيع وصلاحيات العناصر

Signature Options / SigOptions

الـ SigOption: هو كارت أمني أو معرّف أمني يتم إنشاؤه لربطه بعنصر واجهة معين (مثل زر مخصص، حقل، أو تبويبة كاملة) للتحكم في صلاحية تفعيله أو ظهوره.

💡 النتيجة البصرية
سيختفي هذا الزر أو الحقل تماماً عن واجهات كافة موظفي الشركة، ويظهر حصرياً فقط لأعضاء مجموعة مديري الـ IT.

2. محرك الواجهات الشرطية (Conditional UI Engine)

ماذا لو أردنا حماية أعمق؟ مثلاً: جعل حقل الأولوية Priority يتحول لـ Read-Only ومغلق تماماً عن التعديل، ولكن فقط إذا تحولت حالة أمر الشغل إلى CLOSE.

  1. نفتح الـ Advanced Properties للعنصر (Control) المراد حمايته في مصمم الشاشات.
  2. نربط العنصر بـ SigOption (ليقوم النظام بفحص هوية ومجموعة المستخدم أولاً).
  3. نفتح جدول قيود الواجهات الشرطية ونحقن التكوين:
    • تحديد المجموعة الأمنية المصابة بالشرط.
    • اختيار الـ Condition ID (مثل: ISWOSTATUSCLOSE والذي يحمل لوجيك: :status = 'CLOSE').
    • تحديد الخاصية: inputmode تأخذ القيمة readonly (أو visible = false لإخفاء المكون تماماً).
⚠️ آلية التنفيذ في الـ Runtime
يقوم الأبلكيشن سيرفر بتشغيل حلقة فحص متواصلة (Evaluation Loop)؛ بمجرد قيام المستخدم بفتح أمر شغل حالته مغلقة، يستشعر المحرك اللوجيك، ويقوم بحقن الـ CSS/JS الخاص بالقفل ديناميكياً ليتحول الحقل لـ Read-Only فوراً على شاشة العميل.

1. تعريف الـ Cron Task وبنيته الهيكلية

2. التمييز المصيري: التعريف مقابل المثيل

Definition vs Instance

لفصل المنطق البرمجي عن الإعدادات التشغيلية، يقسم المحرك الـ Cron Task إلى جزأين في الجداول:

📘
الـ Cron Task Definition (التعريف)
يمثل المخطط أو الـ Blueprint الأساسي للمهمة. يُكتب فيه اسم كلاس الجافا المسؤول والذي يعمل كامتداد لكلاس psdi.server.CronTask. مثال: كرت تعريف الـ COMMCRONTASK المسؤول عن إرسال البريد الإلكتروني.
⚡
الـ Cron Task Instance (المثيل)
يمثل الوظيفة الحقيقية التي يديرها السيرفر. يمكنك إنشاء أكثر من Instance (نسخة) من نفس الـ Definition الواحد، بحيث تعمل كل نسخة ببارامترات ومواعيد مختلفة تماماً.

3. خصائص وبارامترات الـ Cron Task Instance

Instance Attributes

  • Schedule التعبير الزمني الذي يحدد متى تنطلق المهمة (يدعم صيغة الـ Cron Expression القياسية، مثل 0 0 * * * للعمل كل يوم عند منتصف الليل).
  • Run as User حقل في غاية الأهمية الأمنية؛ يحدد اسم الحساب الأمني (User ID) الذي سيعمل الكود تحت مظلته.
  • Active Checkbox لتشغيل أو إيقاف النسخة.
  • Keep History لتفعيل جدول سجلات التتبع؛ يسجل المحرك تاريخ ووقت وتفاصيل نجاح أو فشل كل دورة تشغيل.
  • Parameters Table جدول متغيرات يختلف حسب كلاس الجافا المستخدم، يتيح حقن قيم نصية للكود (مثل تحديد مسار مجلد أو حجم الدفعات).
🚨 قاعدة أمنية صارمة
كود الجافا عندما يستعلم من الجداول في الخلفية، يخضع تماماً لقيود البيانات وعزل المواقع (Data Restrictions & Sites) الخاصة بالـ Run as User المكتوب هنا. إذا تم اختيار مستخدم صلاحياته فقط على موقع القاهرة، فلن تتمكن الـ Cron Task من قراءة أو تعديل سجلات موقع الإسكندرية أبدًا.

1. مفهوم مسار العمل (Workflow)

هو اللوجيك البرمجي (Software Logic) المسؤول عن تسيير وتوجيه السجل (Routing) عبر محطات العمل. بدلاً من قيام الموظف بالبحث يدوياً في الجداول عن المعاملات التي تحتاج لاعتماده، يتولى الـ Workflow دفع السجل إلى لوحة تحكم الموظف المسؤول تلقائياً، وبمجرد إنجاز الموظف لعمله، ينقل المحرك السجل للخطوة التالية في قاعدة البيانات.

2. توجيه التعيينات للشخص الصحيح (Assignments Routed to the Right Person)

يعتمد المحرك على تكامل فيدرالي مع موديول الـ Roles (الأدوار). يمنع النظام كتابة اسم موظف محدد بنص صريح (Hardcoded User ID) في المسار؛ بل نحدد الـ Role (مثل: مدير الموقع الحالي لأمر الشغل)، وفي الـ Runtime يقوم المحرك بعمل Dynamic Resolution لقراءة داتا السجل، والبحث عن إيميل وحساب الشخص المعني بدقة وتوجيه المهمة لصندوقه.

3. أهداف مسارات العمل (Workflow Goals)

يحدد الكتاب 5 أهداف أساسية للمنصة:

  1. أتمتة وتخصيص عمليات البيزنس (Automate & Customize): تحويل الإجراءات الورقية إلى خطوات رقمية مرنة.
  2. إدارة دورة حياة المستند كاملة (End-to-End Management): تتبع السجلات والتحكم بها من الإنشاء وحتى الأرشفة.
  3. دفع البيانات للمستهدفين بدلاً من السحب اليدوي (Push Information): إعلام المعنيين بالمهام تلقائياً.
  4. توفير سجل مراجعة فيدرالي كامل (Audit Trail): تسجيل الموافقات والرفض وحفظها للحوكمة.
  5. ضمان تنفيذ العمليات وتكرارها بنفس المعايير (Repeatable Practices): تحقيق ثبات مستويات الجودة والأداء.

4. فئات مسارات العمل (Workflow Categories)

ينقسم هذا المحرك برمجياً إلى ثلاث فئات وظيفية حسب طريقة حركة وتفاعل البيانات:

⚙️
مسار العمل الإجرائي القياسي (Process Workflow)
مسار هيكلي صارم وقائم على التعيينات البشرية (Assignment-oriented)؛ يتحكم في دورة السجل، ويفحص الشروط (Conditions)، ويدفع بالمهام لصناديق الموظفين.
⚡
التفاعلات القائمة على السياق (Context-Based Interactions)
مسار خالٍ من التعيينات الفردية (Assignment-less)؛ يقوم ببرمجة سياق تفاعل المستخدم مع الشاشة؛ حيث يعرض للمستخدم "قائمة خيارات ديناميكية" (Action Choices Menu) تتغير ملامحها بناءً على داتا وحالة السجل.
🔄
المسارات الهجينة (Hybrids)
دمج يجمع بين التوجيه الهيكلي التقليدي والتصفح الديناميكي المنبثق للشاشات (مثل وقف الحفظ برمجياً ونقل المستخدم قسرياً لتبويب Failure Reporting لو ترك الخانات فارغة).

5. صندوق الوارد للمهمات (Workflow Inbox / Assignments)

هو المكون البرمجي المصغر (Portlet) المعروض في الـ Start Center للمستخدم. يعرض جدولاً حياً بكافة التعيينات الموجهة للموظف حالياً؛ ومنها يفتح السجل الأصلي لاتخاذ قرار التوجيه الفوري.

6. مكونات ومراحل مسار العمل (Workflow Components & Stages)

Components: يدمج الـ Workflow في بطنه الـ Roles للمستلمين، والـ Actions لتغيير الحالات وحقن القيم، والـ Communication Templates لإرسال الإيميلات، والـ Escalations لمنع اختناق المهام.

🚨 تحذير صارم من الكتاب (Stages)
نظرًا لخطورة الأتمتة الحية، يجب تصميم وتجربة مسارات العمل بالكامل داخل بيئة اختبار معزولة (Test Environment) تحاكي بيئة الإنتاج بدقة، وعند الاستقرار، يتم حزم الـ Workflow ونقله لبيئة الـ Production عبر الـ Migration Manager.

2. تفكيك العقد الثمانية القياسية (Canvas Tab: Node Tool Descriptions)

يتكون شريط الأدوات الرسومي من 8 عقد أساسية، لكل منها سلوك برمجي فريد في الباك إند:

العقدة الوصف البرمجي القوانين والقيود الهندسية الصارمة
🏁 Start Node نقطة انطلاق السجل ليعبر بوابات المسار. يسمح بوجود عقدة بدء واحدة فقط لا غير (Exactly One) لكل مسار عمل.
🛑 Stop Node محطة النهاية وخروج السجل من المسار الحركي. يسمح بوجود أي عدد يشاء المطور من عقد الإيقاف داخل المسار الواحد.
🔀 Condition Node عقدة ذكية تقوم بتقييم تعبير سيكول (SQL WHERE Clause) أو تشغيل كلاس جافا. يشترط خروج خطين اتصال حصرياً ووجوباً: خط اتصال إيجابي (Positive Line تمثل True) وخط اتصال سلبي (Negative Line تمثل False).
📝 Task Node توليد التعيينات البشرية (Assignments) وضخ السجلات داخل الـ Inbox للموظفين. تشترط تحديد الـ Role المسؤول والمهلة الزمنية للتنفيذ.
📥 Manual Input Node تُستخدم عندما يتطلب القرار تدخلاً بشرياً صريحاً للمفاضلة واختيار الخطوة التالية من قائمة خيارات (Choice List). عدد الخيارات يطابق عدد خطوط الاتصال الخارجة منها. أهم قيد أمني: يمنع المحرك تماماً خروج خط اتصال سلبي (Negative Connector)؛ كافة الخطوط الخارجة منها يجب أن تكون إيجابية صلبة.
🔄 Subprocess Node تداخل ومطابقة مسار عمل كامل بالاسم (Nested Workflow Process) ليعمل كخطوة فرعية داخل المسار الرئيسي لتنظيم الكود. تمتلك مخرجات إيجابية وسلبية.
🤝 Interaction Node توجيه تجربة المستخدم ومساعدته عبر فتح شاشة تطبيق معين بالاسم، أو فتح تبويب محدد (Tab Redirection)، أو إظهار نافذة رسائل إرشادية. يسمح بدخول أي عدد من الخطوط إليها، ولكن يشترط خروج خط اتصال واحد فقط لا غير (Exactly One Exit Point) منها.
⏳ Wait Node عقدة تجميد حركية؛ يتوقف المسار عندها وينتظر في حالة سكون إلى أجل غير مسمى. تنتظر السجل حتى يستقبل المحرك إشارة بحدوث "حدث نظام معين" (System Event) لتستيقظ وتمرر السجل فوراً عبر مخرجها الوحيد.

يتلخص مسار البناء في: إضافة العقد على الـ Canvas، ترتيبها، ورسم خطوط الاتصال (Connectors).

Process Duplication: تتيح ميزة الـ Duplicate عمل نسخة كربونية كاملة من المسار الحالي لتبدأ كـ Process جديدة كلياً وباسم مستقل في الداتا بيز.

2. مراجعات العمليات وقوانين النشاط الصارمة (Process Revisions)

يسمح المحرك بوجود أي عدد من نسخ المراجعات (Revisions) لنفس عملية الـ Workflow (مثل Rev 1, Rev 2) لتعديل وتطوير اللوجيك.

🚨 القيد المعماري القاطع للـ Assessment
يسمح بنشاط وتشغيل نسخة مراجعة واحدة فقط بالاسم في نفس الوقت (Only ONE revision can be active at any time) لكل كائن. عند تفعيل المراجعة الجديدة رقم 2، يقوم المحرك تلقائياً بعمل تعطيل (Deactivate) للمراجعة القديمة رقم 1، مع بقاء السجلات الحالية المعلقة داخل المراجعة 1 تعمل حتى تنتهي تماماً دون أي انقطاع في دورتها الحياتية.

1. خصائص العقد والخطوط الإيجابية والسلبية

عند النقر المزدوج على أي عقدة تفتح شاشة خصائصها وضبط روابط الأكواد والـ Actions المرتبطة بها. الخطوط الصادرة من عقد الشرط تنقسم إلى:

  • Positive Connector: خط متصل صلب ويمثل True (الموافقة).
  • Negative Connector: خط منقط ويمثل False (الرفض) لتسيير السجل.

2. الإشعارات وقوالب المتغيرات (Notifications, Roles, & Comm Templates)

مثال استخدام المتغيرات: أمر الشغل رقم :wonum مخصص للمعدة :assetnum

3. ضبط عقد المهام ونوافذ الخصائص (Configuring Task Node Properties)

Assignments window: التبويب الإجباري لعقدة المهمة، ويشترط فيه تحديد Role واحد على الأقل (تعيين بشري للمستلم) لضمان ضخ السجل في الـ Inbox الصحيح. ويمكن ضبط مدد زمنية (Time Limits) لتقييد الموظف بوقت لإنجاز المعاملة.

4. فخ المنطق السلبي مع القيم الفارغة في كود السيكول (SQL NULL Trap)

🚨 أهم تريكة برمجية يركز عليها الكتاب (اختبار قدرتك على كتابة كود SQL متوافق)

المشكلة: تخيل أنك كتبت شرطاً داخل الـ Condition Node لفصل أوامر الشغل التي لا تتبع نوع الصيانة الهامة MI، وصيغت كود الـ WHERE clause كالتالي: :worktype <> 'MI'.

سلوك الباك إند المفخخ: إذا قام مستخدم بإنشاء أمر شغل جديد وترك حقل الـ worktype فارغاً تماماً (NULL)، فكيف سيقيمه المحرك؟ قواعد الـ ANSI SQL تنص على أن مقارنة أي قيمة مع الـ NULL ترجع دائماً نتيجة مكسورة (Unknown/False). بما أن المحرك فشل في إثبات صحة الشرط، سيعتبر النتيجة تلقائياً False، ويقوم بقذف وتوجيه السجل قسرياً عبر الخط السلبي (Negative Connector). هذا سيتسبب في معاملة أمر الشغل العادي كأنه أمر صيانة حرجة بالخطأ وتشتيت التيم.

الحل المعماري الصارم من الكتاب: لمنع فخاخ الـ NULL في المنطق السلبي، يجب صياغة وحقن كود الفحص الصريح للـ NOT NULL دائماً داخل التعبير الشرطي، ليصبح كود الـ WHERE الاحترافي كالتالي:

:worktype <> 'MI' AND :worktype IS NOT NULL

5. القواعد النحوية الصارمة لكود أحداث عقدة الانتظار (Syntax of the Wait Node)

لكي تضبط عقدة الانتظار لكي تستمع لحدث نظام معين، يشترط المحرك كتابة نص الحدث داخل خانة Event باتباع سياق صارم يتكون من 3 أو 4 كلمات باللغة الصغيرة (Lowercase) تفصل بينها نقاط كدليمتيرز:

Syntax: maximo · objectname · eventtype · optionalstatus
  • الكلمة الأولى: ثابتة دائماً وإجبارياً وهي maximo.
  • الكلمة الثانية: اسم كائن البيزنس الملموس المستهدف صريحاً في الـ Metadata (مثل workorder أو sr).
  • الكلمة الثالثة: نوع حركة النظام وهي حصرية بين الخيارات التالية: (init, add, update, delete, أو statuschange).
  • الكلمة الرابعة: (خاصة بحركات الـ statuschange فقط حالياً) وتكتب فيها القيمة الداخلية المشفّرة للحالة المستهدفة في الدومين (مثل wmatl).
💡 مثال المنهج القياسي
maximo.workorder.statuschange.wmatl (المسار يجمد السجل وينتظر في حالة سكون حتى تتحول حالة أمر الشغل لـ Waiting on Material).

📥
Manual Input Node
تطلق شاشة خيارات منبثقة للمستخدم للمفاضلة اليدوية واختيار المخرج. قيدها الصارم: يمنع تماماً خروج خط اتصال سلبي منها؛ كافة مخرجاتها خطوط إيجابية صلبة.

1. مرحلة الفحص والتمكين (Validating, Enabling & Activating)

الـ Validation: عملية موازية تماماً لعملية الـ Compilation في لغات البرمجة؛ حيث يفحص المحرك روابط العقد والـ Syntax للتأكد من وجود مسار متصل ممتد بكامل طول الشاشة دون أي انقطاع هيكلي.

🚨 تنبيه حرج جداً للـ Assessment
نجاح الـ Validation يضمن صحة الـ Syntax والهيكل الفني فقط، ولكنه لا يضمن نهائياً أن لوجيك البيزنس يعمل بشكل صحيح على الأرض.

الـ Enable: رفع علم (Flag) على الميتا-داتا يعلن أنها حزمة صالحة للاستخدام والنشر بعد اجتياز الفحص.

2. قوانين التحقق التسعة الصارمة للـ Compiler (Validation Criteria)

يمنع المحرك تماماً تفعيل أو تمكين أي مسار عمل، ويخرج نافذة أخطاء حمراء إذا تم انتهاك أي بند من القوانين الهيكلية التسعة القياسية التالية التي تحكم الـ Canvas:

  1. يجب على كل عقدة في الشاشة أن يمتلك خط اتصال واحد على الأقل يدخل إليها (يستثنى عقدة البدء Start Node).
  2. يجب أن يخرج من عقدة البدء (Start Node) خط اتصال إيجابي واحد فقط لا غير (Exactly One).
  3. يجب على كل عقدة في الشاشة أن يمتلك خط اتصال واحد على الأقل يخرج منها (يستثنى عقد الإيقاف Stop Nodes).
  4. يجب على كل عقدة مهمة (Task Node) أن يمتلك خط اتصال واحد على الأقل يخرج منها.
  5. يشترط على كل عقدة مهمة (Task Node) أن تحتوي على تعيين بشري واحد على الأقل (At least one Assignment / Role).
  6. يشترط على كل عقدة شرط (Condition Node) خروج خطين اتصال حصرياً ووجوباً كلاهما معاً: خط إيجابي صلب وخط سلبي منقط.
  7. يجب على كل عقدة إدخال يدوي (Manual Input Node) أن يخرج منها خط اتصال إيجابي واحد أو أكثر؛ ويمنع منعاً باتاً خروج خط اتصال سلبي منها.
  8. يشترط على كل عقدة عملية فرعية (Subprocess Node) خروج خطين اتصال معاً: خط إيجابي وخط سلبي حصراً.
  9. يمنع المحرك إنشاء مسار عمل فارغ؛ يجب أن يحتوي المسار على الأقل على عقدة مهمة (Task Node) واحدة معرّفة وصحيحة.

1. الطرق الثلاثة لإطلاق وميلاد الـ Workflow (Initiation Methods)

كيف يدخل السجل (مثل أمر الشغل) داخل شبكة الـ Workflow لأول مرة؟ يقسم الكتاب آليات الميلاد إلى ثلاث طرق حصرية:

🖱️
أ. الإطلاق اليدوي (Manual)

⚡
ب. الإطلاق عبر محرك التصعيد (Escalation-Initiated)
نقوم ببناء كرت Escalation يفتش الجداول كل فترة زمنية، وعند عثوره على سجلات تحقق الشرط، يستدعي كرت Action نوعه APPACTION يحمل أمر الجافا WF Initiate، ليقوم بحقن وقذف كافة هذه السجلات دفعة واحدة داخل الـ Workflow أوتوماتيكياً.
🔄
ج. الإطلاق التلقائي الفوري عند الحفظ (Auto-Initiated)
تفعيل خيار Set Process to Auto-Initiate داخل خصائص المسار; بمجرد قيام أي مستخدم بإنشاء سجل والضغط على "حفظ" (Save)، يستشعر الباك إند الحدث ويقذف السجل فوراً داخل الـ Workflow في نفس ملي ثانية الحفظ.
🚨 قاعدة حاسمة للـ Assessment
يمنع المحرك تفعيل خيار الـ Auto-Initiate لأكثر من مسار عمل واحد فقط كحد أقصى للـ Object الواحد (Limited to one process per object) منعاً لتضارب السيرفر.

✅ الإجراء المعماري للتفعيل
لكي تظهر الأزرار، نطلق إجراء مخصص داخل مصمم الشاشات يسمى Add Workflow to Applications. يقوم المحرك بحقن الميتا-داتا الخاصة بالأزرار في جداول الواجهات، ويقوم أوتوماتيكياً بتوليد وحقن كروت أمنية جديدة (Signature Options / SigOptions) تابعة للـ Workflow داخل تطبيق الـ Security Groups، ليتولى الـ Admin منح صلاحية رؤية واستخدام زر الـ Route للمجموعات المصرح لها.

2. الاستعلام بالمثال (Query by Example - QBE)

المفهوم البرمجي: هو البوابة المباشرة لكتابة استعلام SQL بدون كتابة كود. عندما يكتب المستخدم قيمًا داخل الخانات الفارغة في شاشة الـ List Tab، يقوم المحرك برسم وتوليد شرط WHERE تلقائي بالخلفية لفلترة الجدول.

3. مؤشرات الأداء الرئيسية (Key Performance Indicators - KPIs)

قيمة رقمية مفردة (Single-Value Metric) يتم حسابها وعرضها ديناميكياً لتوضح كفاءة العمليات (مثل: عدد أوامر الشغل المتأخرة). تدعم العرض البياني كأرقام أو رسوم بيانية ملونة (أحمر، أصفر، أخضر) بناءً على حدود مستهدفة (Thresholds).

4. التقارير القائمة على الاستعلام (Query Based Reports - QBR)

وتسمى أيضاً Ad hoc Reports. هي تقارير مرنة يتيح النظام للمستخدمين المتقدمين (Power Users) إنشاؤها وصياغتها وتحديد أعمدتها وتجميعها مباشرة من واجهة الويب دون الحاجة لكتابة كود جافا أو تصميم ملفات خارجية.

5. تقارير المؤسسات باستخدام BIRT (Enterprise Reports using BIRT)

المفهوم المعماري: هي التقارير الرسمية والعملاقة للشركة. يتم تصميمها خارجياً على برنامج Eclipse BIRT (Business Intelligence and Reporting Tools)، وتتميز بقدرتها على معالجة لوجيك معقد جداً، وجداول متداخلة، ورسوم بيانية ضخمة، ويتم تسجيلها وتفعيلها في النظام حصرياً من قِبل الـ Administrator.

  • More Search Fields: نافذة منبثقة متقدمة. التريكة الفنية هنا أنها لا تعرض فقط أعمدة الجدول الرئيسي، بل تعرض أعمدة من جداول أخرى ترتبط بالكائن الحالي عبر الـ MAXRELATIONSHIP (مثل البحث عن أمر شغل بناءً على اسم المورد المعرف في جدول الشركات الخارجي).

3. مثال استعلام أوامر الشغل (A Sample Work Order Query)

يستعرض المنهج شاشات عملية لكيفية دمج الشروط؛ مثلاً فلترة أوامر الشغل التي تقع في موقع BEDFORD وحالتها الحالية تنتظر المواد WMATL عبر خانات الواجهة.

4. مكون وإعداد مجموعة النتائج (Result Set Portlet & Setup)

Result Set Portlet: نافذة مصغرة داخل الـ Start Center تعرض داتا جدولية مباشرة بناءً على استعلام محفوظ.

عند إعداد البورتلت، يجبرك النظام على:

  1. اختيار الاستعلام المحفوظ المستهدف (Saved Query).
  2. اختيار الأعمدة المراد ظهورها بالترتيب (Column Selection).
  3. تحديد سقف لعدد الصفوف المعروضة (Row Limit) لحماية الميموري من الثقل.

5. إنشاء وتتبع الـ KPIs ومراكز البدء

يتيح النظام إنشاء الـ KPI مباشرة من داخل التطبيق (Creating KPIs within an Application) عبر خيار مخصص في قائمة الإجراءات لتسيير الحسابات الرياضية (COUNT, SUM) على الداتا الحالية، وضخها في شاشات الـ Start Centers لمراقبته.

6. فتح التقارير وصفحة الطلب (Accessing Reports & Request Page)

عند الضغط على أيقونة التقارير، تفتح الـ Reports Window، ومنها ننتقل إلى صفحة الطلب (Request Page) وهي الواجهة التي تسبق توليد التقرير ويحدد فيها المستخدم:

  • بارامترات الفلترة (Parameters مثل تاريخ البدء والانتهاء).
  • طريقة فرز وترتيب الأعمدة (Sorting).
  • نوع الملف المخرج.

7. خيارات الجدولة، العرض، والتصدير (Scheduling & Exporting)

  • Scheduling Options: يتيح الـ Request Page جدولة التقرير ليعمل في الخلفية عبر كرت كرون تاست مخصص يسمى REPORTSCHED.
  • Viewing & Exporting: يدعم المحرك عرض التقرير كـ HTML تفاعلي، أو تصديره وحفظه بصيغ ثابتة كـ PDF أو ملفات CSV/Excel لغرض التحليل الخارجي.

🚨 التريكة المعمارية الأهم لمهندسي الباك إند

الحل المعماري (ROS): لكي يظهر تطبيق معين داخل شاشة الـ QBR ويسمح بإنشاء تقارير عليه، يجب أن يكون معرّفاً له مسبقاً Reporting Object Structure (ROS) داخل تطبيق Object Structures.

الـ ROS هي عبارة عن شجرة علاقات آمنة ومحددة سلفاً من قبل المطور (Parent-Child Graph)؛ مثال: هيكل تقارير أوامر الشغل يحدد أن الأب هو WORKORDER والأبناء المسموح بالقراءة منهم فقط هم ⚠️ LABTRANS (الساعات) و MATTRANS (المواد). لا يمكن للمستخدم في الـ QBR الخروج عن نطاق هذه الشجرة المسموحة.

نظرة عامة على إطار التكامل
Integration Framework Overview • 7 محاور رئيسية
▼

1. تعريف التكامل (Definition of Integration)

التكامل في بيئة TPAE يعني القدرة على تبادل البيانات حية الأثر (Transactional Data) أو التأسيسية (Master Data) مع أنظمة خارجية.

💡 مثال عملي
عند تعيين موظف جديد في سيستم الـ HR (مثل Workday)، نريد حقنه تلقائياً في جدول PERSON داخل TPAE دون إدخال بشري يدوي.

2. معمارية إطار التكامل (The Integration Framework)

الـ MIF هو موديول مدمج بالكامل يعيش داخل الأبلكيشن سيرفر (MXServer). يعتمد كلياً على معايير الـ SOA (Service Oriented Architecture) ويستخدم لغة الـ XML أو الـ JSON كصيغة موحدة (Payload Standard) لنقل وهيكلة البيانات.

3. الميزات التقنية الرئيسية (Integration Framework Key Features)

⚡
نوعي الاتصال (Connectivity Mode)
يدعم الاتصالات الفورية المتزامنة (Synchronous) والاتصالات الخلفية المؤجلة غير المتزامنة (Asynchronous).
🔄
تطويع البيانات (Transformation)
يمتلك محركات مدمجة لترجمة البيانات أونلاين عبر Processing Rules أو استخدام ملفات الـ XSL Stylesheets لتبسيط الرسالة للسيستم الخارجي.
🔌
تعدد البروتوكولات (Protocols)
يدعم الشحن عبر الـ REST APIs، الـ Web Services (SOAP)، قنوات الـ HTTP، الملفات النصية المسطحة (Flat Files)، وجداول الاستضافة (Interface Tables).

4. حزمة تطبيقات التكامل (Integration Framework Application Set)

تدار هذه المنظومة بالكامل من داخل موديول الـ Integration في القائمة الشجرية، وتضم 5 تطبيقات مترابطة:

  1. Object Structures: بناء وتصميم هيكل البيانات.
  2. Publish Channels: قنوات الصادر التي ترسل الرسائل من النظام للخارج.
  3. Enterprise Services: خدمات الوارد التي تستقبل الرسائل من الخارج وتصبها بالنظام.
  4. External Systems: تعريف وبيانات الأنظمة الشريكة التي يتم التكامل معها.
  5. Endpoints: تحديد نهايات خوادم الاستقبال وعناوين الوجهة وقنوات النقل.

5. أنواع قنوات التكامل (Integration Types)

ينقسم تسيير الرسائل برمجياً إلى نوعين حسب تدفق المعالجة والوقت:

  • Inbound (الوارد): رسائل قادمة من الخارج لحقن داتا في جداول TPAE، وتمر إلزامياً بمحرك الـ Validation التابع للـ MBO لحماية الداتابيز من الحشو الخاطئ.
  • Outbound (الصادر): رسائل تنطلق من TPAE نحو الخارج، ويتم إطلاقها عبر الـ (Event Listeners) بمجرد ضغط "حفظ" أو تغيير حالة في الشاشات.

6. هياكل الكائنات (Object Structures)

المفهوم البرمجي: هي اللبنة الأولى وحجر الأساس للنظام؛ وتعمل تماماً كالـ DTO (Data Transfer Object) أو الـ Serializer. هي عبارة عن رسم شجري منطقي (Graph) يجمع كائن بيزنس واحد أو أكثر تجمعهم علاقات Parent-Child. وعلتها تحويل البيانات العلاقية المعقدة في قاعدة البيانات إلى ملف XML موحد وسهل القراءة.

7. تطبيق الـ Object Structures

الواجهة التي نحدد فيها شجرة الكائنات (مثل هيكل MXASSET الذي يربط جدول ASSET كـ Parent مع جدول ASSETMETER كـ Child).

🚨 تريكة الأداء وتحسين الحجم (Select/Exclude)
تتيح لك شاشة Object Structures إجراء Select / Exclude Fields؛ حيث يمكنك استبعاد الأعمدة الحساسة أو غير المطلوبة (مثل حقول كلمات المرور أو حقول النظام الداخلية) من الرسالة لتقليص حجم ملف الـ XML وتوفير باندويدث الشبكة.

DEBUG يكتب كل شيء بالتفصيل الممل (استعلامات السيكول، حركات الكائنات، فحص الـ MBO) - يستهلك الميموري بكثافة.
INFO يكتب الأحداث العامة العادية (إقلاع السيرفر، دخول الموظفين، بدء المهام الخلفية).
WARN يكتب التحذيرات التي لا تعطل السيستم ولكنها تمثل خطورة مستقبلاً (مثل قرب انتهاء رخصة أو تجاوز حدود ميموري معينة).
ERROR يكتب الأخطاء الصريحة والـ Exceptions التي ضربت كود الجافا (مثل فشل حفظ سجل أو تعطل اتصال بقاعدة البيانات).
FATAL الكوارث الحرجة التي تتسبب في توقف الموديول أو سقوط الـ Thread كلياً واستحالة الاستمرار.
✅ إجراء تطبيق الإعدادات (Apply Settings Action)
بعد تغيير مستوى اللوج لأي موديول (مثل تحويل موديول الـ maximo.crontask إلى DEBUG لصيد مشكلة في مهمة خلفية)، نضغط على زر Apply Settings من قائمة الإجراءات. يقوم المحرك بعمل Live Refresh في الـ RAM فوراً لتبدأ اللوجات الجديدة بالظهور على الهارد ديسك في نفس الثانية.

4. إدارة الـ Appenders ونافذة التحكم

الـ Appender: هو المعرف البرمجي الذي يحدد "أين ومكان شحن وكتابة اللوج" (Log Destination Path). يتيح النظام إنشاء وإدارة الـ Appenders من نافذة مخصصة تسمى Manage Appenders Window، وتنقسم إلى ثلاثة أنواع حصرية في المعمارية:

🖥️
Console Appender
يوجه كود الجافا ليطبع اللوجات مباشرة على شاشة كونسول السيرفر الأسود الفوري.
📁
File Appender
يوجه اللوجات لتُكتب وتُحفظ داخل ملف نصي مسطح صريح على هارد ديسك السيرفر.
🔄
Rolling File Appender
ملف نصي يمتلك سقفاً لحجم الملف (Max File Size مثل 10MB). بمجرد امتلاء الملف، يقوم المحرك أوتوماتيكياً بأرشفته وتسميته برقم (مثل maximo.log.1) ويفتح ملفاً جديداً كلياً باسم maximo.log للاستمرار، مما يحمي الهارد ديسك من الامتلاء المفاجئ.
🛠️

1. أساسيات وتتبع أعطال المكونات (Basics & Component Troubleshooting)

عند حدوث مشكلة، تنص ممارسة الـ Isolation على مراجعة سلامة المكونات التحتية بالترتيب:

2. إدارة خدمات ومدخلات الـ IBM WebSphere

تقع ملفات خدمات وسكريبتات الـ WebSphere افتراضياً في مسار مجلد الـ Profile الخاص بالخادم:

C:\IBM\WebSphere\AppServer\profiles\CtgAppSrv01\bin

التحكم عبر الـ Command Prompt: يتيح للمطور إطلاق الأوامر التنفيذية التالية يدوياً لإدارة وإقلاع السيرفرات:

  • startServer.bat MXServer: لإقلاع وبدء تشغيل مثيل خادم التطبيقات الرئيسي المضيف للمحرك.
  • stopServer.bat MXServer: لإيقاف الخادم وإنهاء الجلسات بأمان.

3. حاويات الويب وأعراض مشاكلها (Web Containers & Symptoms)

الـ Web Container: هو الجزء الداخلي في خادم التطبيقات المسؤول عن استقبال طلبات الـ HTTP القادمة من متصفحات المستخدمين، ومعالجة صفحات الـ JSPs والـ Servlets، وتوجيه الـ Traffic.

🚨 أعراض وعلامات كوارث الـ Web Container
  • ظهور رسائل خطأ صفحة الويب الشهيرة HTTP 500 Internal Server Error أو HTTP 404.
  • حدوث بطء وتهنيج كامل في استجابة الواجهات نتيجة تعرض الـ Thread Pool للاختناق (Exhaustion)؛ حيث يمتلئ السيرفر بطلبات مستخدمين معلقة لا تجد Thread حر لمعالجتها.
  • ظهور كوابيس الـ java.lang.OutOfMemoryError: Java heap space نتيجة تسريب في الميموري (Memory Leaks).

4. خريطة ودليل أماكن ملفات اللوجات (IBM WebSphere Log Files & Locations)

عند سقوط السيرفر أو حدوث خطأ غامض، يشترط المنهج على مهندس الصيانة حفظ ومعرفة المسارات الفيزيائية الصارمة لملفات اللوجات التالية على السيرفر لقراءتها فوراً:

اسم ملف اللوج المسار الفيزيائي الافتراضي على السيرفر الوظيفة البرمجية والبيانات المخزنة داخله
SystemOut.log ...\profiles\CtgAppSrv01\logs\MXServer\ اللوج المركزي الأهم للسيرفر؛ تصب فيه كافة رسائل الـ INFO والـ DEBUG وأوامر طباعة الكود وجلسات الموظفين الحية.

5. أخطاء ملفات الخصائص (Properties Errors)

يستعرض الكتاب أخطاء مطابقة ملف الـ maximo.properties؛ مثل: قيام المطور بتغيير باسورد الداتا بيز يدوياً وترك القيمة بنص صريح دون تشغيل سكريبت الـ encryptproperties.bat، أو عدم مطابقة كود الـ mxe.db.schemaowner لاسم مالك المخطط الحقيقي في DB2، مما ينتج عنه فشل الخادم في الإقلاع كلياً وكتابة قيد خطأ اتصال في ملف SystemErr.log.
